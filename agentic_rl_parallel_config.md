# Qwen3-235B-A22B Agentic RL：长序列并行配置寻优（128k / 1M · 256 die）

> 目标：为 **128k** 与 **1M** 两种上下文窗口，在 **N=256 die** 下按不同优化目标（最大吞吐 / 均衡 / 并发·显存优先）给出并行配置。
> 关键约束（用户设定）：
> - **长序列必开 SP/CP**：SP(=TP，0 额外占卡) 切残差激活；CP 切序列维（激活 + KV）。
> - **超长序列（1M）不开大 TP**：TP 限在单 UB 岛内（≤8），序列缩放靠 **CP**，而非把 TP 拉大跨岛。
> - **用 ZeRO-2，不用 ZeRO-3**：只切 梯度+优化器，**参数不按 DP 切**（省掉逐层 param all-gather）；专家参数本就被 EP 切小，ZeRO-3 的额外省显存不值那笔通信。
> - per-die HBM 预算：**A2 64 / A3 64 / 950DT 72 GB**（A2 die = A3 die）。判定按 **64GB/die**（保守，A3 通用），950DT 多 8GB 余量。

---

## 1. 模型与建模口径

**模型**：Qwen3-235B-A22B（MoE）· 235B 总 / 22B 激活 · 94 层 · h=4096 · 128 专家 top-8 · GQA · 专家参数 ~227B、非专家 ~8B。

**并行分解**：`N = TP × CP × PP × DP = 256`（单位 die）；`EP = CP × DP`（专家沿 CP×DP 分片、TP 组内复制 ETP=1）；`SP = TP`。

### 1.1 per-die 显存公式（ZeRO-2）

```
训练阶段峰值 = 静态 + 激活
  静态(ZeRO-2) = 专家权重 454/(EP×PP)        # fp16, 沿 EP 切、TP 内复制
               + 非专家权重 16/(TP×PP)        # fp16
               + 梯度+优化器 ~13              # (2+12)B/param, 分布式切到全部 256 die
  激活        = 101 × (S/128k) / (TP×CP)      # 残差流+激活重算, SP=TP 与 CP 共同切序列

推理阶段峰值 = 底座 + KV
  底座(FP8)  = 227/EP + 8/TP
  KV/die     = 25.2 × (S/128k) × b / (TP×CP)  # b=每组并发序列数

单 die 峰值 = max(训练阶段, 推理阶段)  ≤  64 GB
```

**为什么 ZeRO-2 就够**：专家权重已被 EP 切到 7–14GB；ZeRO-3 再沿 TP 复制维切参数只多省 ~10GB，却要**每层 param all-gather**——在长序列已被 CP/TP 通信占满的情况下不划算。ZeRO-2 只切梯度+优化器（~13GB，无逐层 all-gather），参数留复制。

### 1.2 两场景的裸量（未分片，单序列）

| 量 | 128k | 1M | 随 S |
|---|---|---|---|
| 训练激活（残差流） | 101 GB | 808 GB | ∝ S |
| KV / 序列 | 25.2 GB | 201.6 GB | ∝ S |
| 注意力计算 | 基准 | ×64 | ∝ S² |
| 静态（参数+梯度+优化器） | ~30 GB | ~30 GB | 不随 S |

→ 长序列的显存全压在**激活与 KV**上，且二者都 ∝ 1/(TP×CP)；**SP/CP 是唯一杠杆**，这正是"长序列必开 SP/CP"的原因。

## 2. 128k 场景寻优（N=256）

激活 = 101/(TP×CP)；KV/die = 25.2·b/(TP×CP)。128k 下显存不紧，配置差异主要在**吞吐 ↔ 并发(b) ↔ 通信**权衡。

| 目标 | 配置（EP=CP×DP，SP=TP） | 训练阶段 | 推理阶段 | b | 峰值 | 特点 |
|---|---|---|---|---|---|---|
| **吞吐优先 T1** | `TP8·SP8·CP1·PP1·DP32`（EP32） | 29+12.6=**42** | 8+50=**59** | 16 | 59 ✅ | **无 CP**、DP32 最大吞吐、TP8 落 UB 岛 |
| **均衡 B1** | `TP8·SP8·CP2·PP1·DP16`（EP32） | 29+6.3=**35** | 8+50=**59** | 32 | 59 ✅ | CP2 换来 **b=32**（2× rollout 并发） |
| **并发/显存 C1** | `TP8·SP8·CP4·PP1·DP8`（EP32） | 29+3.2=**32** | 8+25=**33** | 32 | 33 ✅ | 余量极大，可推 **b→64**；DP8 吞吐低 |

**读法**：128k 三代 die（64GB）都轻松放下。**从 T1→C1**：CP 增大（跨节点通信增）、DP 减小（吞吐降），换来更大的 b（rollout 并发）和更多余量。**默认推 T1**（吞吐优先，无 CP）；若要高 rollout 并发选 B1。

## 3. 1M 场景寻优（N=256）

激活 = 808/(TP×CP)；KV/die = 201.6·b/(TP×CP)。**必须 TP×CP ≥ ~32** 才放得下（放下训练激活需 TP×CP≳20，放下推理 KV@b8 需 ≥28）。TP 限 ≤8（单岛），其余靠 CP。

| 目标 | 配置（EP=CP×DP，SP=TP） | 训练阶段 | 推理阶段 | b | 峰值 | 特点 |
|---|---|---|---|---|---|---|
| **吞吐优先 T2** | `TP8·SP8·CP4·PP1·DP8`（EP32，TP×CP=32） | 29+25=**54** | 8+50=**59** | 8 | 59 ✅ | **CP 最小(4)**、TP 填满岛 → 跨节点通信最少 |
| **均衡 B2** | `TP4·SP4·CP8·PP1·DP8`（EP64，TP×CP=32） | 24+25=**49** | 5.5+50=**56** | 8 | 56 ✅ | **TP 小(4)**、CP8 承担缩放（贴合超长序列惯例） |
| **并发/显存 C2** | `TP4·SP4·CP16·PP1·DP4`（EP64，TP×CP=64） | 24+12.6=**37** | 5.5+50=**56** | 16 | 56 ✅ | TP×CP=64 → 支持 **b=16**；CP16 通信最重、DP4 吞吐最低 |

**读法**：
- **T2（吞吐优先）**：把 TP 开到岛上限 8、CP 压到最小 4。理由：**TP all-reduce 走岛内 UB（快），CP all-to-all 跨节点（慢）**——同样的 TP×CP=32，多 TP 少 CP 通信更省。
- **B2（均衡，贴合"小 TP"惯例）**：TP=4、CP=8。跨节点 CP 通信比 T2 多，但 TP 更小、更符合"超长序列不开大 TP"的工程惯例；吞吐略低于 T2。
- **C2（并发/显存优先）**：TP×CP=64 把激活/KV 再切一半，**支持 b=16**（2× rollout 并发）；代价是 CP=16 跨节点通信最重、DP=4 训练吞吐最低。
- **可选 PP**（仅训练阶段）：`TP4·CP8·PP2·DP4` 可把每 stage 层内存再 /2，缓解 1M 训练激活；但 PP 不进 rollout、两阶段需重排，仅当训练阶段仍紧时才加。

## 4. 选型建议与跨场景对照

| 维度 | 128k | 1M |
|---|---|---|
| 显存是否紧 | 松（峰值 33–59 / 64） | **紧**（56–59 / 64，必须 TP×CP≥32） |
| 主导旋钮 | b（并发） | **CP**（序列缩放） |
| 默认推荐 | **T1**`TP8·CP1·DP32`（吞吐） | **T2**`TP8·CP4·DP8`（吞吐，CP 最小） |
| 高并发选 | B1`TP8·CP2·DP16` b=32 | C2`TP4·CP16·DP4` b=16 |
| 贴合"小 TP" | — | B2`TP4·CP8·DP8` |

### 4.1 三个结构性结论
1. **CP 是 1M 的命门，且抬带宽诉求**：CP 的序列 all-to-all/ring ∝ S·h，1M 下 CP=4–16 的**跨节点通信量巨大** → **必须超节点 fabric（A3/950DT）**，A2（8-die 岛装不下 EP 域 + 扛不住 CP 跨节点）**彻底不可行**。
2. **TP 上限=岛大小（8），缩放交给 CP**：TP>8 跨岛得不偿失；用满 TP=8 再加 CP 是通信最省的路径（T2）。
3. **decode 被 CP 兜住**：KV/die 被 TP×CP 切到 ~25–50GB → decode 每步 KV 读 ~15–32ms（per-die 1.6TB/s）；**FP8 KV 可再减半**，是 1M rollout 的关键杠杆。

### 4.2 硬件与卡数
- N=256 die → 物理卡：**A2 256 卡 / A3 128 卡 / 950DT 128 卡**（A3/950DT 双 die）。
- HBM 判定按 **64GB/die**（A2=A3）；950DT die 72GB 多 ~8GB 余量、且 HBM BW 2.0TB/s 使 decode 更快。
- **A2 全表不可行**（scale-up fabric 装不下 EP 域 + CP 跨节点）；**A3 可行、950DT 从容**。

## 附录 · 符号与公式

**符号**：`N`=die 数=256 · `TP/CP/PP/DP`=张量/上下文/流水/数据并行(die) · `EP=CP×DP` · `SP=TP` · `S`=序列长 · `b`=每组并发序列 · `h`=4096 · `L`=94 · 精度 训练 BF16 / 推理权重 FP8。

**核心公式**（per-die）：
```
N = TP×CP×PP×DP = 256          EP = CP×DP          SP = TP
静态(ZeRO-2)  = 454/(EP×PP) + 16/(TP×PP) + 13        # 专家wt + 非专家wt + 梯度优化器
训练激活      = 101 × (S/128k) / (TP×CP)
推理 KV/die   = 25.2 × (S/128k) × b / (TP×CP)
底座(FP8)     = 227/EP + 8/TP
单die峰值     = max(静态+激活, 底座+KV) ≤ 64 GB
scaling       激活∝S · KV∝b·S · 注意力∝S² · CP↑→跨节点带宽↑
```
