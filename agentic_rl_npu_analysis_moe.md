# Agentic RL 后训练：Qwen3-235B-A22B 在华为 A2/A3/950DT 上的逐场景资源充分性分析

> 模型 **Qwen3-235B-A22B**（MoE：235B 总参 / **22B 激活**，94 层，hidden=4096，**128 专家 top-8**，**GQA** 64Q/4KV，FP8 可用；多模态用 **Qwen3-VL-235B-A22B**）· 算法 **GRPO** · 初始请求 8k · global batch = **1024 prompt** · mini batch = **32 prompt** · **共卡 colocated · on-policy**
> 本文把 agentic RL 后训练流水线的逐边（A–G）传输/存储量代入华为三代 NPU（A2=910B、A3=910C、950DT），在**文本/图片/视频**三场景下判断**卡间带宽、HBM、CPU DRAM** 是否够用。数字均为**量级级 sizing**，假设已参数化，替换即可重算。
> **本文核心**：Qwen3-235B-A22B 是 MoE 模型——其**权重同步/优化器/训练内部通信由总参 235B 驱动**（而非激活 22B），且训练内部主导通信是**专家并行 all-to-all（随 S 线性涨）**。这两点决定了它在 NPU 上的瓶颈画像（见 §1、§8）。

## 1. 方法与判定口径

判定口径：把每条边折算成**单卡·每迭代**量 ÷ 链路带宽，判是否 comm-bound；HBM 按 rollout/training 两相取峰值；DRAM 按单节点 8 卡核算。**Qwen3-235B-A22B 作为 MoE 模型，有三处关键特性，决定了本文的分析重点：**

| 量 | Qwen3-235B-A22B 的特性 |
|----|------------------------|
| **权重同步（边 A）/ 优化器 / 显存里的权重** | ∝ **总参 Φ_total=235B**——不是激活 22B！任一 token 可路由到任一专家，**全部 128 个专家都要同步/驻留** |
| **计算（FLOPs、注意力）** | ∝ **激活 Φ_act=22B**——每 token 只算被选中的 **8/128** 专家 |
| **训练内部主导边（边 G）** | **专家并行 all-to-all**（dispatch+combine 激活）——**∝ tokens=B×S，随 S 线性涨** |

- **✅符号**：✅ 富余 · ⚠️ 紧张/需优化 · ❌ 放不下/需换配置。
- **精度**：Qwen3-235B-A22B 有官方 FP8 权重，本文默认权重/计算可走 **FP8**（边 A、显存权重减半），优化器仍 FP32。

> **一句话直觉**：MoE 用"总参大、激活小"换容量——**省的是计算，不省通信与显存里的权重/优化器**。所以 Qwen3-235B 在 NPU 上的瓶颈偏向**权重搬运、优化器内存、专家 all-to-all**，而非算力。

## 2. 硬件规格：A2（910B）vs A3（910C）vs 950DT

| 规格 | **A2 · 910B** | **A3 · 910C** | **950DT**（2026 Q4） | 对本文的影响 |
|------|--------------|---------------|----------------------|----------------|
| 单卡 HBM | **64 GB** | **128 GB** | **144 GB** | 装总参分片 + 优化器 + KV |
| HBM 带宽 | ~1.6 TB/s | ~3.2 TB/s | ~4 TB/s | — |
| scale-up 互联 | **HCCS ~392 GB/s，仅节点内 8 卡** | **UB ~392 GB/s，超节点 ≤384 卡** | **灵衢 ~2 TB/s/卡，超节点 ≤8192 卡** | ★★ **MoE 的 all-to-all 对此更敏感**（见 §5） |
| scale-out | RoCE ~25 GB/s/卡 | RoCE ~50 GB/s/卡 | 灵衢统一编址 | A2 超 8 卡即掉到 RoCE |
| 低精度算力 | BF16 ~376 TFLOPS | BF16 ~800 TFLOPS | FP8 1P / FP4 2P FLOPS | MoE 激活小、算力不是瓶颈 |
| 超节点规模 | 8 卡岛 | ≤384 卡 | ≤8192 卡 | **决定 EP all-to-all 能否留在高速域** |

> 数字为代表值，以目标集群实测为准。**scale-up fabric 的"可达范围"是本文的胜负手**：MoE 的专家 all-to-all 是**二分带宽（bisection）敏感、全体互发**的通信模式，极吃 fabric 的"全连接"能力——A2 的 8 卡岛尤其吃亏（见 §5）。

## 3. 并行与部署配置（MoE colocated on-policy）

**专家必须走 EP**。**共卡下 rollout 与 training 分时复用同一批 128 卡，但两相的并行 mesh 不同**（尤其注意力的切法）。

| 共用项 | 配置 | 说明 |
|--------|------|------|
| 总卡数 `N` | **128** | A3：单超节点全 UB；A2：16 节点×8，跨节点 RoCE |
| 专家并行 `EP` | **32**（128 专家 → **4 专家/卡**，训练/推理共用同一套专家分片） | 专家不移动，token 经 all-to-all 送到专家所在卡 |
| 同步 | **on-policy**，每迭代边 A 同步 1 次 | 走本地 scale-up |
| 精度 | 权重/计算 **FP8** 可选；训练优化器 FP32 | Qwen3-235B-A22B-FP8 |

### 3.1 训练侧并行（EP 专家 + ZeRO-3 其余）

| 组件 | 并行方式 | 每卡承担 |
|------|---------|---------|
| **专家 FFN**（227B，占总参 ~97%） | **EP=32** | 4 专家/卡 = 7.1B 参数（BF16 **14.2 GB** / FP8 7.1GB）；专家**不移动**，靠 all-to-all 把 token 送来 |
| **注意力(GQA)+embed+router**（~8B） | **ZeRO-3（全分片，DP=128）** | 参数分片到 128 卡、用时 all-gather；常驻仅 **~0.1 GB** |
| **梯度**（BF16） | ZeRO-3 /128 | 470GB/128 ≈ **3.7 GB** |
| **优化器状态**（FP32×3 = 2.82TB） | ZeRO-3 /128 | **22.0 GB** |
| **层**（94） | PP=1（基线；长 S 可 PP>1 缓解激活） | — |
| **批** | 数据并行（EP 组间 + ZeRO DP） | 每 optimizer step 本卡 512/128×… 见 §5 |

- 训练静态合计 ≈ **40 GB/卡**（§3.3）。**主导通信 = 专家 all-to-all（边 G）** + 非专家的 ZeRO all-gather（小）。
- 注意力用 ZeRO-3 而非 TP：省常驻、便于优化器统一分片。

### 3.2 推理侧（rollout）并行（EP 专家 + TP 注意力 + DP）

**推理 mesh = `EP(32) × TP(4) = 128`**。同一个"32"维**双重角色**：对专家是 **EP=32**（128 专家 → 4/卡，全体经 all-to-all 共享）；对注意力是 **DP=32**（32 条数据并行 lane，每 lane 内 TP=4）。即注意力 = **DP=32 × TP=4**。

| 组件 | 并行方式 | 每卡承担 |
|------|---------|---------|
| **专家 FFN** | **EP=32**（与训练同一分片布局） | 4 专家/卡，FP8 ~7 GB |
| **注意力(GQA)** | **DP=32 × TP=4**（4 KV head → 1/卡） | ★ **KV cache 按 TP 切 /4**——降单卡 KV 与解码延迟（§6 的 KV÷4 即来源于此） |
| **权重** | EP+TP 分片（FP8） | ~9 GB/卡 |
| **KV cache** | 随并发 × S | 0.79 / 1.58 / 6.3 GB/序列/卡（§6） |
| **优化器 / 梯度** | **无**（推理不需要） | 释放 ~26GB 给 KV cache |

- **主导通信 = 生成时每 MoE 层的专家 all-to-all**（dispatch+combine，逐 token/step，延迟敏感）。
- 推理用 TP=4 而非 ZeRO：把 KV cache 切成 /4（GQA 4 KV head 正好 1/卡），显著降单卡 KV——这是推理相区别于训练相的关键。
- **DP=32** 决定了并发序列/prompt 如何分摊到卡：边 B 的"1024 prompt ÷ 32 组"、以及 rollout 并发的分布都源于此。

### 3.3 单卡静态量（训练阶段 与 推理阶段）

**训练阶段**（EP 专家 + ZeRO-3 其余 + 优化器）：

```
总参 235B → 权重 BF16 470 GB / FP8 235 GB；
优化器(FP32×3) = 235B×12B = 2.82 TB

专家参数分片(EP=32)：  227B routed / 32 × 2B ≈ 14.2 GB/卡（FP8 ~7.1）
非专家(ZeRO-3 /128)：  ~0.1 GB/卡（常驻；用时 all-gather）
梯度(BF16, ZeRO/128)： 470/128 ≈ 3.7 GB/卡
优化器(ZeRO/128)：     2.82TB/128 ≈ 22.0 GB/卡
────────────────────────────────────────
训练静态合计 ≈ 40 GB/卡（BF16 权重）/ ~33 GB（FP8 权重）  ＋ 激活（∝S，见 §6）
```

**推理阶段**（rollout：EP 专家 + TP=4 注意力，**无优化器/梯度**）：

```
专家权重(EP=32, FP8)：   227B / 32 × 1B ≈ 7.1 GB/卡（BF16 ~14.2）
非专家权重(TP=4, FP8)：  ~8B / 4 × 1B ≈ 2.0 GB/卡（attention+embed）
────────────────────────────────────────
推理"权重底座" ≈ 9 GB/卡（FP8）/ ~18 GB（BF16）  ＋ KV cache（∝并发×S，见 §6）
```

> **两相对照与共卡峰值**：
> - **训练相**吃内存的是**优化器 22GB**（总参 235B 的 FP32 状态），权重底座只是零头——静态 40GB。
> - **推理相**没有优化器/梯度，权重底座仅 ~9GB（FP8），**省下的 ~30GB 全给 KV cache**。
> - **共卡分时**：任一时刻显存峰值 = max(训练相, 推理相)。文本/低并发时**训练相（40GB+激活）主导**；视频/高并发时**推理相（9GB+KV cache）主导**（视频 KV 可爆到上百 GB，见 §6）。

### 3.4 训练 vs 推理：共用什么、差什么

| | 训练相 | 推理相（rollout） |
|---|--------|-------------------|
| 专家 | EP=32（4/卡） | EP=32（**同一分片布局**） |
| 注意力 | **ZeRO-3**（DP=128，KV 不跨卡切） | **DP=32 × TP=4**（KV 切 /4） |
| 内存构成 | 参数分片 + 梯度 + 优化器 ≈ 40GB | 权重分片 + KV cache（**无**优化器/梯度） |
| 主导通信 | 专家 all-to-all + 反向 + ZeRO all-gather | 专家 all-to-all（逐 token 解码，延迟敏感） |

- **共用专家分片布局**是关键：边 A 权重同步只需把更新后的**专家分片（EP=32 对齐）**刷进推理副本，分片感知、无需重排（见 §5 边 A）。
- **共卡分时**：rollout 时释放梯度/优化器（或 offload）给 KV cache；training 时释放 KV cache 给激活。峰值取两相较大者。

### 3.5 流水线与边（MoE 视角）

**共卡域**：整条流水线跑在**同一批 128 NPU 上**（rollout 与 training 分时复用）；右侧 **(A) 权重同步**是训练引擎 → Rollout 引擎的回环，走**本地 scale-up**。图为**一次 RL 迭代**；边旁数字为**单卡·每迭代**量（文本/图片/视频三值）。**训练内部主导通信 = 边 G**：训练引擎（及 rollout 生成）内部每个 MoE 层的 **专家 all-to-all（dispatch↔combine）**，随 tokens=B×S 增长。

```
      ┌──────────────────────────────────────────────────┐
      │ 调度器 / 数据集                                  │
      └──────────────────────────────────────────────────┘
        | (B) prompt 分发   ~1 MB/卡
        v
      ┌──────────────────────────────────────────────────┐
      │ Rollout 引擎 (EP=32 + TP=4, GQA)                 │<──────────────┐
      │ 每 prompt 采样 G=16 条                           │               │
      │ 生成: 每 MoE 层做 专家 all-to-all                │               │
      └──────────────────────────────────────────────────┘               │
        | (C) 轨迹回传(D2H): id + 压缩帧   8MB / 0.4GB / 1.4GB/卡        │
        v                                                                │
      ┌──────────────────────────────────────────────────┐               │
      │ 经验缓冲区 (host DRAM, 分片存)                   │               │
      └──────────────────────────────────────────────────┘               │
        | 读轨迹: 打分 & 算 old/ref logprob                              │  (A) 权重同步  训练 -> Rollout
        v                                                                │       总参 470GB / FP8 235GB (每卡分片 ~7GB)
      ┌──────────────────────────────────────────────────┐               │       迭代末尾 x1, 走本地 scale-up
      │ 打分 + 算 old/ref logprob (读 -> 回填)           │               │
      │ (D) 奖励/优势    (E) old/ref logprob             │               │
      └──────────────────────────────────────────────────┘               │
        | (F) 取 mini-batch(H2D, 含帧供 ViT 重算)  8MB / 0.4GB / 1.4GB   │
        v                                                                │
      ┌──────────────────────────────────────────────────┐               │
      │ 训练引擎 (EP=32 专家 + ZeRO-3 其余)              │               │
      │ optimizer step 内层循环 (x32)                    │               │
      │ 每 MoE 层: dispatch <-> combine = (G) all-to-all │               │
      │ (G) ~52 / 103 / 413 TB/卡 (文本/图片/视频)       │───────────────┘
      └──────────────────────────────────────────────────┘
```

**读图要点：**
- **边 G（本文主角）不在框外、在框内**：dispatch↔combine 发生在训练引擎（及 rollout 生成）**每个 MoE 层内部**，是全体互发的 all-to-all，**∝ tokens=B×S、随 S 涨**（~52/103/413 TB/卡）——这是本流水线训练内部的绝对主导通信。
- **(A) 权重同步搬的是总参**（470GB/FP8 235GB），即便只有 22B 激活也要全同步；共卡下走本地 scale-up，每卡只刷自己那份专家分片（~7GB）。
- **边 B/C/D/E/F 都是 MB 级**、走 host/PCIe，任何代际都是零头；**唯一带宽敏感的是框内的边 G**（走 scale-up，见 §5）。

## 5. 单卡逐边（A–G）通信量 → 带宽是否够

### 5.1 单卡逐边通信量表（每迭代，N=128）

**代入值（下表所有符号，先定义后使用）：**
- **模型**：`Φ_total`=235B（权重 BF16 470GB / FP8 235GB）；`Φ_nonexpert`≈8B（注意力+embed+router）；`h`=4096；`n_layers`=94；`top_k`=8。
- **batch**：`N_p`=1024（prompt 数）；`G`=16（每 prompt 采样数）；`B`=`N_p×G`=16,384（轨迹数）；`mini`=32；`n_steps`=`N_p/mini`=32。
- **序列**：`P`=8k；`S`=16k / 32k / 128k（文本/图片/视频）；`R`=`S−P`=8k / 24k / 120k。
- **并行**：`N`=128（总卡）；`EP`=32；`TP`=4（推理注意力）；`n_grp`=`N/TP`=32（prompt 分发的 TP 组数）。
- **字节**：权重/激活 BF16=`2B`；token id、logprob=`4B`。
- `n_mb`：梯度累积 micro-batch 数（长 S 增多，见下"关键公式说明"）。

| 边 | 内容 | 计算公式（单卡·每迭代） | 单卡量（文本/图片/视频） | 链路 | 随 S? |
|----|------|--------------------------|--------------------------|------|------|
| **A** | 权重同步 | 每卡驻留权重分片 ≈ `Φ_total / EP`（专家按 EP=32 分片）= 235GB/32(FP8) | **~7 GB**（FP8；BF16 ~14GB） | scale-up | 否 |
| B | prompt 分发 | `N_p×P×4B / n_grp`（TP 组内复制，n_grp=32） | ~1 MB | host | 否 |
| C | 轨迹回传(D2H)：token id **+ 压缩帧** | `B×S×4B/N`(id) **+ 压缩帧字节** | 8 MB / **~0.4 GB** / **~1.4 GB** | PCIe | 帧主导 |
| D | 奖励+优势 | `B×8B / N` | ~1 KB | host | 否 |
| E | old+ref logprob(D2H) | `2×B×R×4B / N` | 8 / 25 / 126 MB | PCIe | 线性(小) |
| F | mini-batch 取数(H2D，**含帧供 ViT 重算**) | 同边 C（累计） | 8 MB / **~0.4 GB** / **~1.4 GB** | PCIe | 帧主导 |
| **G** | **EP all-to-all**（训练 fwd+bwd） | `(B×S/N)·n_layers·2·top_k·h·2B · 2` | **~52 / 103 / 413 TB** | scale-up | **是（∝S）** |
| — | 额外前向(old+ref) 的 all-to-all | `(B×S/N)·n_layers·2·top_k·h·2B · 2` | **~52 / 103 / 413 TB** | scale-up | 是（∝S） |
| — | 非专家参数 all-gather（attention/GQA） | `~3·Φ_nonexpert·2B·n_mb` | 数 TB（小） | scale-up | 弱 |
| | **scale-up 单卡合计** | 主导=all-to-all | **~105 / 210 / 830 TB** | | 是 |
| | **PCIe 合计（C+E+F）** | | ~24 MB / **~0.9 GB** / **~2.9 GB** | | 帧主导 |

**关键公式说明：**
- **边 C/F 的多模态大头是"压缩帧"，不是 token id**：方案① 下 buffer 存的是**压缩原始帧**（图片 ~8 图×400KB ≈ 3.2 MB/轨迹；视频 数十帧压缩片段 ≈ ~10 MB/轨迹），训练侧再重跑 ViT。所以边 C（rollout→buffer）与边 F（buffer→训练，帧要 H2D 供 ViT 重算）对图片/视频**由帧字节主导**：每卡 = `B×帧字节/N` ≈ **图片 0.4 GB、视频 1.4 GB**，而 token id 数组（`B×S×4B` = 17/67 MB）反倒是零头。帧量假设可参数化（换分辨率/帧数需重代入）。**但仍是 PCIe 级零头**（视频 ~1.4GB 走 ~50GB/s 仅 ~28ms），不改带宽判定；边 E（logprob，无帧）不受影响。这也与 §7 经验缓冲区"视频 ~十几 GB/节点"（帧聚合量）对齐。
- **边 A（MoE 税）**：即便只有 22B 激活，**全部 235B 专家都要同步**（任一 token 可能路由到任一专家）。聚合 = 470GB(BF16)/235GB(FP8)；单卡按专家分片（EP=32）后 ~7–14GB，反而不大——大头是聚合量，不是单卡量。
- **边 G = EP all-to-all（本文主角）**：每个 MoE 层，token 的 hidden（`h×2B`）要 dispatch 给它选中的 `top_k=8` 个专家、再 combine 回来 → 每 token 每层 `2·top_k·h` 元素。乘 94 层、乘本卡 token 数 `B×S/N`、乘 (fwd+bwd)：
  - 文本 `(268M/128)·94·(2·8·4096·2B)·2 ≈ **52 TB/卡**`；加 old/ref 两次额外前向再 ~52TB → **~105 TB/卡**。
  - **∝ tokens=B×S，随 S 线性**：图片 ~210TB、视频 ~830TB/卡。
- **边 G ~105TB/卡起步且随 S 涨**——量级更大、且是 **all-to-all（二分带宽敏感、全体互发）**，对 fabric 的"可达范围+全连接带宽"要求远高于 all-gather。

### 5.2 带宽判定：all-to-all 打在 scale-up fabric 上

**EP 组内全体互发**，一旦 EP 跨出高速域就极易崩溃。

| | scale-up（EP=32 是否在高速域） | 文本 ~105TB 纯通信 | 判定 |
|---|---|---|---|
| **950DT** | 灵衢超节点，EP=32 全在域内，~2TB/s + 16.3PB/s 二分带宽 | ~50 s | ✅✅ 专为 all-to-all 设计 |
| **A3** | UB 超节点 ≤384 卡，EP=32 全在域内，~392GB/s | ~270 s | ✅（文本/图片，overlap 成立但**偏紧**）/ ⚠️（视频 ~2100s，二分带宽吃紧） |
| **A2** | HCCS 仅 8 卡岛；**EP=32 跨 4 节点 → all-to-all 走 RoCE ~25GB/s** | ~4200 s | ❌❌ all-to-all 无法跨 RoCE，彻底崩 |

### 5.3 带宽判定结论（代际主导）
- **A2 ❌❌**：EP all-to-all 是"全体互发、同步、二分带宽敏感"，一旦超出 8 卡 HCCS 岛就落到 RoCE 二分带宽，彻底崩。除非把整个 EP 组塞进 8 卡（EP≤8 → 每卡 16 专家、显存吃紧且专家并行度不足），否则 A2 跑不动 235B MoE 的 RL。
- **A3 ✅/⚠️**：单超节点内 EP=32 全走 UB，文本/图片 overlap 成立（但绝对通信量大、余量小，强依赖 all-to-all/计算重叠与 dispatch-combine kernel 优化）；视频 all-to-all ~830TB/卡，二分带宽吃紧，判 ⚠️。
- **950DT ✅✅**：灵衢的 16.3 PB/s 聚合二分带宽正是为 all-to-all/大 EP 设计，三场景从容，且可上更大 EP/world。

## 6. 单卡 HBM 存储 → 显存是否够

Qwen3 用 **GQA**（64 Q / 4 KV head）；显存的特殊点在**总参驱动的静态量**与**长 S 的激活/KV**。

### 6.1 KV cache（GQA，随 S 线性）

`V_kv = 2×L(94)×h_kv(512)×S×2B`；TP=4 → 每卡 `/4`：

| 场景 | S | 每卡每序列 KV |
|------|---|---------------|
| 文本 | 16k | 0.79 GB |
| 图片 | 32k | 1.58 GB |
| 视频 | 128k | 6.30 GB |

### 6.2 单卡 HBM 峰值与判定

共卡时，**训练阶段**与**推理阶段**分别核算，峰值取二者较大。

- **训练阶段**（静态 40GB + 激活）。激活按一个 micro-batch 只有一条序列计算：`≈ n_layers×S×h×2B`（存层边界残差流）：

| 场景 | 静态 | 激活(1 序列, 开重计算) | 合计 | A2(64) | A3(128) | 950DT(144) |
|------|------|----------------------|------|--------|---------|------------|
| 文本 | 40 | `94×16k×4096×2B` ≈ **12.6** | **~53 GB** | ✅(紧) | ✅ | ✅ |
| 图片 | 40 | ≈ **25.2** | **~65 GB** | ❌(略超) | ✅ | ✅ |
| 视频 | 40 | ≈ **101** | **~141 GB** | ❌ | ❌ | ⚠️(逼近144) |


- **推理阶段**（推理权重 EP+TP 分片 FP8 ~7GB + KV cache，b=32/组）：

| 场景 | 权重(FP8) | KV(b=32) | 合计 | A2(64) | A3(128) | 950DT(144) |
|------|-----------|----------|------|--------|---------|------------|
| 文本 | 7 | 25.3 | **~32 GB** | ✅ | ✅ | ✅ |
| 图片 | 7 | 50.6 | **~58 GB** | ⚠️(逼近64) | ✅ | ✅ |
| 视频 | 7 | 201.6 | **~209 GB** | ❌ | ❌ | ❌ |

### 6.3 HBM 判定小结（共卡峰值 = max(训练阶段, 推理阶段)）
- **静态训练态 40GB**：三代都放得下（A2 ✅）——总参 235B 经 EP+ZeRO 分片后，单卡训练静态可控。
- **文本**：训练阶段 ~53GB、推理阶段 ~32GB → A2 紧但可、A3/950DT ✅。
- **图片**：训练阶段 ~65GB 已略超 A2(64)、推理阶段 ~58GB 逼近 → **A2 ⚠️/❌**（需减激活/降并发），A3/950DT ✅。
- **视频**：两相都撑不住——训练阶段激活 ~101GB 需序列并行/PP，推理阶段 KV ~200GB 需 offload+降并发；**三代都得动并行/offload**，A2 基本不可行、A3 勉力、950DT 余量最大。

## 7. CPU host DRAM 是否够

按单节点 8 卡、~1.5 TB DRAM 核算（方案① 编码）。

| 来源 | 量级（每节点） | 说明 |
|------|----------------|------|
| 经验缓冲区（token id + logprob **+ 压缩帧**，方案①） | 文本 ~2GB / 视频 ~十几 GB | 视频那部分主要是**压缩帧**（∝帧数×分辨率），非 token id |
| 优化器 offload（**可选**） | 2.82TB / 16 节点 ≈ **176 GB/节点** | N=128 时优化器本可留 HBM（22GB/卡）；A2 若显存吃紧可 offload |
| 视频 KV offload | b=8 时 ~50GB/卡 × 8 ≈ 400 GB/节点 | 视频高并发时 |
| Agentic 工具沙箱 | 数十~数百 GB（需压测） | 不可解析，coding 沙箱可达 TB，按工具实现压测 |

**判定**：方案① 下经验缓冲区不是大头；主要变量是**优化器是否 offload（176GB/节点）**与**视频 KV offload（~400GB/节点）**。合计通常在**数百 GB/节点**，1.5TB **✅ 够用**。MoE 的优化器虽大（2.82TB），但 N=128 分摊后单节点可控。

## 8. 结论

格式 `A2 / A3 / 950DT`：

| 资源维度 | 文本 (16k) | 图片 (32k) | 视频 (128k) |
|----------|-----------|-----------|-------------|
| **带宽（边 G=EP all-to-all）** | ❌❌ / ✅ / ✅✅ | ❌❌ / ✅ / ✅✅ | ❌❌ / ⚠️ / ✅ |
| **HBM（训练相=静态40+激活）** | ✅ / ✅ / ✅ | ❌ / ✅ / ✅ | ❌ / ❌ / ⚠️（需序列并行/PP切激活） |
| **HBM（rollout 峰值 b=32）** | ✅ / ✅ / ✅ | ⚠️ / ✅ / ✅ | ❌ / ❌ / ❌（都需降并发+KV offload） |
| **CPU DRAM（1.5TB/节点）** | ✅ / ✅ / ✅ | ✅ / ✅ / ✅ | ✅ / ✅ / ✅ |

**三条主结论：**

1. **带宽是头号瓶颈，且被 EP all-to-all 决定。** 边 G 是专家并行的 all-to-all（dispatch+combine 激活），**~105 TB/卡起、随 S 线性涨（视频 ~830TB）**，且是"全体互发、同步、二分带宽敏感"的模式。它要求 fabric 不仅"可达"还要"高二分带宽全连接"：**A2 的 8 卡 HCCS 岛容不下 EP=32（跨节点走 RoCE，all-to-all 彻底崩，❌❌）；A3 的 UB 超节点能把 EP=32 收在高速域（文本/图片 ✅、视频 ⚠️）；950DT 的灵衢 16.3PB/s 二分带宽最从容（✅✅）。235B MoE 的 RL 事实上要求超节点级 fabric——至少需要 A3/950DT。**

2. **内存的硬约束是"总参驱动的权重/优化器"与"长 S 的激活/KV"两块。** 权重（470GB BF16/235GB FP8）、优化器（FP32×3=2.82TB）都按**总参 235B**（非激活 22B）算——需要 EP+ZeRO 把它们分片到足够多卡（N=128 下训练静态 ~40GB/卡，三代都放得下）。真正随场景恶化的是长 S：训练激活视频单序列就 ~101GB（需序列并行/PP 切）、推理 KV 视频 b=32 就 ~200GB/卡（需 offload+降并发）——**视频两相都得动并行/offload,三代都吃紧。**

3. **算力不是瓶颈,通信与内存才是。** 每 token 只激活 22B,前反向/注意力的 FLOPs 需求不高;235B MoE 在 NPU 上的全部压力压在 **all-to-all 通信、权重同步、优化器/激活/KV 内存**上,而非算力——所以选型看的是 **fabric 二分带宽 + HBM 容量 + 超节点规模**,不是 TFLOPS。

> **一句话**：**235B MoE 的 agentic RL,成败在 fabric——EP all-to-all 是主瓶颈,只有超节点(A3/950DT)扛得住,A2 8 卡岛出局**;内存上总参驱动的权重/优化器靠 EP+ZeRO 分片可控,但长 S(视频)的激活与 KV 逼着三代都上序列并行/PP + offload;算力全程不是约束。三场景里,文本/图片在 A3/950DT 上可行、A2 不行,视频则要 A3 勉力、950DT 才从容。

## 9. 方法论边界与适用性

- **本文针对 Qwen3-235B-A22B**——当前主流的 agentic RL 后训练 MoE 模型。分析聚焦其"总参驱动内存 + 专家 all-to-all"两大特性在 NPU 上的落地。
- **并行配置（EP=32 专家；训练 ZeRO-3 / 推理 TP=4 注意力）是代表性选择**：MoE 必须用 EP；但绝对 all-to-all 量强依赖 EP 度数、N、dispatch/combine kernel 与计算-通信重叠优化，本文取代表值，换配置需重代入。
- **数字为量级 sizing**：all-to-all 量对 `N`、`EP`、`top_k`、`h` 敏感（∝ `B×S·top_k·h·n_layers/N`）；硬件规格（尤其 A2 RoCE、950DT 灵衢二分带宽）为代表值，以目标集群实测为准。
- **未展开项**：auxiliary-loss-free 负载均衡对 all-to-all 分布的影响、专家激活不均导致的 all-to-all 长尾、Qwen3-VL 视觉编码器自身的并行、YaRN 长上下文的额外开销——这些会改变常数因子，但不改"all-to-all 主导 + fabric 可达范围是胜负手"的定性结论。
