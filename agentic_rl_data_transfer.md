## 1. 流水线架构与阶段划分

GRPO 后训练系统由**推理引擎（Rollout）**、**经验缓冲区（Experience Buffer）**、**奖励/验证器**、**训练引擎（Actor 更新，兼做 logprob 计算）**、**参考模型（KL 项）** 构成。**Rollout 采样完立刻把原始轨迹（未打分）写入经验缓冲区；奖励打分、旧/参考 logprob 计算都是「从缓冲区读数据、算完再写回缓冲区」的后处理步骤，不是 rollout 直接产出的下一环**——这是本图相对早期版本调整的关键顺序。整张图是**一次 RL 迭代（outer loop，执行 1 次）**；训练引擎内部嵌套了一个 **optimizer step 内层循环（inner loop，执行 32 次）**——(G) 的 13.5 TB/GPU 就是这 32 次单 step 通信量（每次 423 GB/GPU）累加出来的。每条边 (A)–(G) 标注方向与单次触发的传输量。

```
        ┌───────────────────────┐
        │   调度器 / 数据集       │
        └───────────────────────┘
                    │  (B) prompt 分发            33.5 MB
                    ▼
        ┌───────────────────────┐  (B') 环境/工具观测    ┌──────────────┐
        │      Rollout 引擎      │ ───────────────────▶ │  环境 / 工具  │
        │    (vLLM / SGLang)    │ ◀─────────────────── │ (code/search)│
        │  每 prompt 采样 G=16 条 │   (拼回上下文, 抬高 S) └──────────────┘
        └───────────────────────┘◀──────────────────────────────────────────┐
                    │  (C) 原始轨迹 token/mask 回传（未打分） 1.07 GB           │
                    ▼                                                       │
        ┌─────────────────────────────────────────────┐                     │
        │         经验缓冲区 (Experience Buffer)        │                     │
        │   先存 C 的原始轨迹；再被下面两路读取、回填打分结果 │                     │
        └─────────────────────────────────────────────┘                     │
              │ 读轨迹token去打分          │ 读轨迹token去算logprob             │
              ▼                          ▼                                  │
   ┌────────────────────┐   ┌────────────────────────────────────┐          │
   │   奖励模型 / 验证器  │   │ 训练引擎（同一份权重）算旧/参考logprob  │          │
   └────────────────────┘   └────────────────────────────────────┘          │
              │ (D) 奖励+优势回填 ~128 KB    │ (E) 旧/参考logprob回填 536.9MB×2 │
              └──────────────▶ 经验缓冲区 ◀──┘                                │  (A) 权重同步（迭代末尾，×1）
                                     │                                      │  Φ×2B = 141 GB
                                     │  (F) 切成 32 个 mini-batch            │  (回环: 训练 → Rollout)
                                     │      (每个 = 32 prompt×G=16=512 条轨迹)│
                                     ▼                                      │
   ┌──────────────────────────────────────────────────────┐                 │
   │   optimizer step 内层循环 (inner loop, ×32)           │                 │
   │                                                      │                 │
   │   取 1 个 mini-batch（512 条轨迹，本地 ≈33.5 MB）       │                 │
   │                │                                     │                 │
   │                ▼                                     │                 │
   │   ┌───────────────────────┐                          │                 │
   │   │    训练引擎 (Actor)    │ ◀── (G) 单 step 通信      │─────────────────┘
   │   │    FSDP / ZeRO-3      │      423 GB/GPU          │          
   │   └───────────────────────┘   (all-gather×2 +        │          
   │                │                reduce-scatter×1)    │          
   │                └──── 更新参数分片, 回到下一个 mini-batch │          
   │                       （循环 32 次后退出）              │          
   └──────────────────────────────────────────────────────┘ 
        32 步累计 (G) = 32 × 423 GB/GPU ≈ 13.5 TB/GPU
```

> **注：图中标注的具体数值（边 C=1.07GB、边 E=536.9MB、边 A=141GB、边 G=13.5TB 等）都是 §2 定义的基线参数下的值**（`S=16k`、`Prompt 8k + Response 8k`、响应全部为生成 token、纯文本）。换场景后部分边会变：长序列见 §6.1，group size / mini-batch 见 §6.2/§6.3，多模态见 §9.1–9.4，**coding agent（长上下文 + `response_mask`）见 §9.5**。其中边 A（权重同步，只取决于 Φ）和边 D（稀疏标量奖励）跨场景基本不变，边 C/E（随 S、mask 变）会明显不同——这是「基线 vs 场景变体」的正常关系，不是数值冲突。

**阶段（stage）与阶段间的「边（edge）」：**

| 边 | 方向 | 传输内容 |
|----|------|----------|
| A | 训练引擎 → Rollout 引擎 | **模型权重同步** |
| B | 调度器 → Rollout 引擎 | 采样用的 prompt token |
| B' | 环境/工具 ↔ Rollout 引擎 | 多轮 agentic 的观测（工具返回、环境反馈） |
| C | Rollout 引擎 → 经验缓冲区 | 生成轨迹的原始 token id、mask（**尚未打分**，先落盘） |
| D | 经验缓冲区 ↔ 奖励模型/验证器 | 缓冲区把轨迹 token 供给验证器打分，验证器算完标量奖励、group 归一化优势后**写回**缓冲区 |
| E | 经验缓冲区 ↔ 训练引擎（logprob 计算） | 缓冲区把轨迹 token 供给训练引擎侧做 teacher-forcing 前向，算完旧策略 logprob + 参考策略 logprob（GRPO KL 项）后**写回**缓冲区 |
| F | 缓冲区 → 训练引擎 | mini-batch 取数（本地/同机，通常不跨网）——取的是已经被 D/E 回填过奖励和 logprob 的完整样本 |
| G | 训练引擎内部 | 前向/反向的参数 all-gather 与梯度 reduce-scatter |

## 3. 模型规模拆解：为什么权重同步是 141 GB

Llama-3-70B 单层参数（GQA：64 Q head / 8 KV head，SwiGLU，`i`=28672）：

| 组件 | 计算 | 参数量 |
|------|------|--------|
| q_proj | 8192×8192 | 67.1 M |
| k_proj / v_proj | 8192×1024 ×2 | 16.8 M |
| o_proj | 8192×8192 | 67.1 M |
| gate / up / down | 8192×28672 ×3 | 704.6 M |
| **单层合计** | | **855.6 M** |
| ×80 层 | | 68.4 B |
| embedding + lm_head（未 tie） | 128256×8192 ×2 | 2.1 B |
| **总计 `Φ`** | | **≈ 70.5 B** |

- **BF16 权重体量** = `70.5e9 × 2 B` = **141 GB** → 这是每次权重同步（边 A）的基础量，**与 batch 大小无关**。
- FP32 主权重 282 GB、Adam 一/二阶矩 2×282 GB、梯度 141–282 GB —— 这些是**显存占用**，常驻训练侧、不跨阶段传输。

## 4. 逐边数据传输量

### 边 A — 权重同步（训练 → Rollout）★ 主导项

每次迭代（on-policy）需把更新后的 Actor 权重刷进推理引擎：

```
V_A = Φ × bytes_param = 70.5e9 × 2 = 141 GB / 迭代
```

- **FP8 同步**：≈ **70 GB**（推理引擎支持 FP8 权重时）。
- **拓扑影响**：
  - *同机 colocated*（veRL 等，训练与推理共用 GPU）：走 NCCL broadcast / CUDA IPC，本地 NVLink，141 GB 在 ~数百 GB/s 下 ≈ 0.3–1 s。
  - *分离部署 disaggregated*（训练集群 ↔ 推理集群）：141 GB 每步跨集群 IB/以太网，是**头号瓶颈**。
- **频率**：严格 on-policy 每迭代 1 次；若允许 off-policy（每 N 步同步一次）则摊薄为 `141/N`。

### 边 B — Prompt 分发（调度器 → Rollout）

只需把 **1024 个 prompt 各发一份**，由 Rollout 引擎在采样时对每条 prompt 各生成 `G=16` 条轨迹（无需重复发送 prompt 本身）：

```
V_B = N_p × P × 4 B = 1024 × 8192 × 4 = 33,554,432 B ≈ 33.5 MB
```
可忽略。

### 边 B' — Agentic 多轮环境/工具交互

多轮 rollout 中，工具返回（代码执行结果、检索文档、环境状态）会**拼回上下文**。字节量本身不大（文本），但**显著抬高 `S`**，从而放大边 C/E 的传输与训练算力。设平均 `k` 轮、每轮观测 `o` token：

```
S = P + Σ(生成_i + 观测_i)   →  R 随轮数线性增长
```
这是 agentic 场景相较普通 RLHF 的关键放大因子（见 §6 敏感性）。

### 边 C — 原始轨迹回传（Rollout → 缓冲区，未打分）

Rollout 采样完成后**先把原始轨迹落盘到缓冲区**，此时还没有奖励、没有 logprob——只传 **token id + attention mask**，不传激活/logits。轨迹总数 `B = N_p × G = 16,384`：

```
总 token 数 = B × S = 16,384 × 16,384 = 268,435,456 ≈ 268.4 M tokens
V_C = 268.4M × 4 B (int32 id) = 1,073,741,824 B ≈ 1.07 GB   (+ mask 约同量级或更小)
```

### 边 D — 奖励与优势（缓冲区 ↔ 奖励模型/验证器，读取后写回）

验证器从缓冲区**读取**边 C 落盘的轨迹 token 进行打分，算完后把标量奖励、优势**写回**缓冲区（即缓冲区里同一条轨迹记录被原地补全）。GRPO 在数学/代码任务多用**规则验证器**，奖励是每条轨迹一个标量；优势在**组内**（同一 prompt 的 G=16 条轨迹）做均值/方差归一化后广播回每条轨迹：

```
V_reward = B × 4 B = 16,384 × 4 = 65,536 B = 64 KB
V_adv    = B × 4 B = 64 KB          （组内归一化优势，按序列广播）
总计 ≈ 128 KB
```
可忽略。若用 **神经奖励模型（另一个 70B）**：输入即边 C 的 token（同量级），输出标量极小；但 RM 前向是**算力**开销，激活不跨阶段。

### 边 E — 旧策略 / 参考策略 logprob（缓冲区 ↔ 训练引擎侧，读取后写回）

训练引擎（及参考模型）从缓冲区**读取**边 C 落盘的轨迹 token，做 teacher-forcing 前向算出 logprob，再**写回**缓冲区。只在**响应 token**上计算每 token 的选中 token 对数概率（FP32）：

```
响应 token 数 = B × R = 16,384 × 8192 = 134,217,728 ≈ 134.2 M
V_old_logp = 134.2M × 4 B = 536,870,912 B ≈ 536.9 MB   （π_old，用于重要性比值）
V_ref_logp = 134.2M × 4 B ≈ 536.9 MB                   （π_ref，GRPO 的 KL 惩罚项）
```

GRPO 需要**参考模型的额外一次全量前向**（算力，见边 G 附加项）+ 上述 536.9 MB 参考 logprob 跨阶段传输。

> **`π_old` 前向是否必须？—— 一个可选但有代价的优化。**
> Rollout 引擎生成时为了采样，本身就会算出每个生成 token 的 logits/logprob，理论上可以直接复用这份 `log π_old`，省掉训练引擎侧那次专门的全量前向（§4 边 G 附加项里的 4.5 TB/GPU）。但多数生产级框架（veRL、OpenRLHF 等）**默认仍然重算**，原因是：
> 1. 推理引擎（vLLM/SGLang，常用 PagedAttention + 低精度/FP8）与训练引擎（FSDP，BF16/FP32）数值路径不同，同一份权重算出的 logits 存在细微漂移；
> 2. GRPO/PPO 的 `ratio = π_new/π_old` 一旦分母掺入「引擎数值噪声」而非纯粹的「策略漂移」，在 clip range 较窄时会让 clipping 判断失真；
> 3. agentic 多轮轨迹里工具/环境注入的 token 并非模型采样产生，天然没有生成时 logprob 可复用，仍需一次 teacher-forcing 前向补齐。
>
> **`π_ref` 前向则没有这条捷径**：参考模型是冻结的初始/SFT 权重，从第一次策略更新之后就与生成轨迹用的 `π_θ` 不同，其 logprob 不可能是 rollout 生成的副产物，必须专门起一次前向——这是 GRPO KL 项的刚性成本，不像 `π_old` 前向那样可选。

### 边 F — mini-batch 取数（缓冲区 → 训练）

通常经验缓冲区与训练引擎同机/同进程，取 mini-batch 走本地内存，**一般不计入跨网传输**。每迭代分 `n_steps=32` 个 mini-batch 取出，每个 mini-batch（32 prompt×G=16=512 条轨迹）≈ `512×16384×4B ≈ 33.5 MB`；32 个 mini-batch 累计 = 总量 `μ × V_C` ≈ `μ × 1.07 GB`，与边 C 总量一致（μ=1 即整批数据只被取用一遍）。

### 边 G — 训练内部集合通信（FSDP/ZeRO-3）★ 训练侧主导

一次优化器 step 的**带宽最优**每卡通信量 ≈ `3Φ`（前向 all-gather 参数 `Φ` + 反向 all-gather 参数 `Φ` + 梯度 reduce-scatter `Φ`）：

```
每 step 每卡 = 3 × Φ × 2 B = 3 × 141 GB = 423 GB   （BF16；若梯度 FP32，reduce-scatter 段翻倍；与 batch 大小无关，只取决于 Φ）
n_steps = μ × N_p / mini = 1024 / 32 = 32
每迭代每卡 = n_steps × 423 GB = 32 × 423 GB ≈ 13.5 TB / GPU
```

- 该「每卡 ~3Φ」在环形/树形集合通信下**近似与 GPU 数 N 无关**；**聚合上网流量**则 ≈ `N × 3Φ`，随 N 线性增长。
- **关键点**：单 step 通信量只取决于 `Φ`，与该 step 处理多少条轨迹（即 `mini × G`）无关——FSDP 每步都要把完整参数 all-gather 齐、把完整梯度 reduce-scatter 掉，这个量纲只跟模型大小挂钩。所以 `n_steps` 完全由 `N_p/mini` 决定，**与 group size G 无关**（见 §6.2）。
- **额外前向**（不进入优化器 step，仅 all-gather，无反向、无 reduce-scatter），按 `N_p/mini = 32` 个 micro-step 计：
  - 计算 `π_old` logprob：一次全量前向 ≈ `(N_p/mini) × Φ × 2B` = 32 × 141 GB ≈ **4.5 TB/GPU**。
  - GRPO 参考模型前向（KL 项）：再 **4.5 TB/GPU**。
  - 两者合计 **≈ 9.0 TB/GPU**，与优化器 step 本身的 13.5 TB/GPU 属同一数量级，**GRPO 的参考模型开销不可忽略**。

## 6. 敏感性分析

固定 `Φ`=70.5B、BF16、μ=1，仅列受影响项。

### 6.1 序列长度 `S`（agentic 多轮的核心变量）

| S (P+R) | 边 C 轨迹 | 边 E old/ref logprob（各一份） | 说明 |
|---------|-----------|-------------------------------|------|
| 16k (P8k+R8k) | 1.07 GB | 536.9 MB | 默认 |
| 32k (P8k+R24k) | 2.15 GB | 1.61 GB | 5 轮左右 agentic |
| 64k (P8k+R56k) | 4.29 GB | 3.76 GB | 长程 agent |

> 权重同步（141 GB）与 `S` 无关，故 `S` 增大时跨阶段流量占比里权重同步的主导性会**逐步下降**；但训练内前向随 `S` 线性、注意力算力随 `S²` 上升。

### 6.2 group size `G`（固定 `N_p=1024`、`mini=32` prompt 不变）

`n_steps = N_p/mini = 32` **不随 `G` 变化**（因为 mini batch 按 prompt 数切分，与每个 prompt 采样几条无关），所以边 G（optimizer 通信）在这张表里是常数；只有边 C/E（经验数据）随 `G` 线性增长：

| G | B（轨迹） | 边 C | 边 E(old, 单份) | n_steps | 边 G/卡（优化器通信，与 G 无关） |
|---|-----------|------|-----------------|---------|-----------------------------------|
| 8 | 8,192 | 537 MB | 268.4 MB | 32 | 13.5 TB |
| 16（默认） | 16,384 | 1.07 GB | 536.9 MB | 32 | 13.5 TB |
| 32 | 32,768 | 2.15 GB | 1.07 GB | 32 | 13.5 TB |

> `G` 越大，组内优势估计越稳定，rollout 生成量、经验回传量（C/E）、单 step 内的 FLOPs 均线性增长；但 optimizer **通信次数与每次通信量不变**（只取决于 `Φ` 和 `N_p/mini`），权重同步 141 GB 也不受影响。

### 6.3 mini batch 大小（固定 `N_p=1024` 不变，单位为 prompt 数）

`mini` 越小，`n_steps` 越多，边 G 总量线性增长（每步仍是固定的 423 GB/GPU）：

| mini（prompt） | 每 mini-batch 轨迹数（×G=16） | n_steps = N_p/mini | 边 G/卡 |
|----------------|-------------------------------|---------------------|---------|
| 16 | 256 | 64 | 27.1 TB |
| 32（默认） | 512 | 32 | 13.5 TB |
| 64 | 1024 | 16 | 6.77 TB |

> mini batch 越大，每步喂入的数据越多、单卡显存压力越大，但换来更少的 optimizer step、更低的通信总量——这是显存与通信量之间的经典权衡。

### 6.4 同步精度

| 精度 | 边 A |
|------|------|
| BF16 | 141 GB |
| FP8 | ~70 GB |

### 6.5 同步频率（off-policy 容忍）

| 每 N 步同步 | 边 A 摊薄 |
|-------------|-----------|
| N=1（on-policy） | 141 GB/迭代 |
| N=4 | 35 GB/迭代 |

## 7. CPU（host）内存峰值

跨阶段/训练内部通信之外，还有一块常被忽略的资源约束：**CPU 内存**。它不跟 GPU 显存共享，一旦打满会直接 OOM-kill 训练进程。峰值高度依赖**是否做了 CPU offload**，分场景给出量级。

### 7.1 不做 CPU offload（纯 GPU 训练，FSDP/ZeRO-3 数据全在显存）

CPU 内存只装“轻量”的东西：

| 用途 | 量级 |
|---|---|
| 经验缓冲区（token id + logprob + reward，即边 C+D+E 之和） | ~2.15 GB |
| Dataloader / prompt 数据集缓存 | 几百 MB ~ 几 GB |
| NCCL pinned memory 通信缓冲区 | 通常几百 MB |
| Python / 框架进程本身开销 | 几 GB |

**总量级：个位数到十几 GB**，不是瓶颈。

### 7.2 优化器状态 CPU offload（ZeRO-Offload / DeepSpeed stage-3 offload）★ CPU 内存暴涨的主因

Adam 优化器状态（FP32 主权重 + 一阶矩 + 二阶矩，梯度视配置可选）搬到 CPU：

```
FP32 主权重: Φ×4B = 282 GB
一阶矩 m:    Φ×4B = 282 GB
二阶矩 v:    Φ×4B = 282 GB
（梯度若也 offload）: Φ×4B = 282 GB
────────────────────────────
合计: 846 GB ~ 1.1 TB
```

ZeRO-3 下这部分是**分片**到所有节点 CPU 上的，单节点 CPU 内存峰值 ≈ `846GB / 节点数`：

| 节点数（每节点 8 卡） | 单节点 CPU 内存峰值（仅优化器状态部分） |
|----------------------|------------------------------------------|
| 2 | ~423 GB |
| 4 | ~212 GB |
| 8 | ~106 GB |
| 16 | ~53 GB |

节点数少（如 1–2 台）时，单机 CPU 内存需求可达**数百 GB 到 1 TB+**，会成为部署的硬约束，需要大内存机型（如 1–2 TB host RAM）。

### 7.3 Rollout 引擎（vLLM/SGLang）的 CPU KV cache offload

agentic 长序列场景常开 CPU offload 存长上下文 KV cache。单条 KV cache：

```
V_kv = 2(K+V) × L × h_kv × S × 2B(BF16)
```

对 Llama-3-70B（GQA，KV head=8 → h_kv=1024）、`S=16k`：

```
V_kv = 2 × 80 × 1024 × 16,384 × 2B ≈ 5.37 GB / 条
```

并发几十到上百条长序列同时 offload 到 CPU 时，量级可达**几百 GB**（如 64 条并发 ≈ 344 GB）。

### 7.4 Agentic 工具调用引入的 CPU 内存 ★ 与 offload 无关的独立来源

前三节的内存需求都能从 `Φ`、`B`、`S` 等模型/batch 参数**解析算出**；但 agentic 场景特有的**工具调用**，其内存需求由**工具执行的并发进程数与单进程内存**决定，与模型大小无关，更像是运维层面需要压测确定的变量，而不是解析量：

| 工具类型 | 单实例内存 | 主要驱动因素 |
|---------|-----------|-------------|
| 代码执行沙箱（Python 解释器/容器，可能加载 numpy/pandas 等库） | ~100–500 MB（空载），加载数据/建模后可到 GB 级 | rollout 引擎的**并发上限**（远小于总轨迹数 B），不是 B 本身 |
| 本地检索索引（FAISS/BM25，供搜索工具用） | 几十 GB（小语料）~ 几百 GB（网页级语料） | 语料规模，**常驻**、不随并发变化 |
| headless 浏览器（网页浏览工具） | ~100–500 MB / 实例 | 并发实例数 |
| 工具返回结果缓存 | 通常与边 B'/C 同量级（文本） | 若返回结构化大数据（整表、图片）会显著更大 |

**估算示例**：假设 rollout 引擎设置 512 路并发，其中 30% 触发代码沙箱（平均 300 MB/个），另加载一个 50 GB 的检索索引：

```
代码沙箱: 512 × 0.3 × 300 MB ≈ 46 GB
检索索引（常驻）: 50 GB
────────────────────────
合计 ≈ 96 GB
```

**关键区别**：优化器/KV cache 的 offload 内存需求由 `Φ`、`S` 等**确定性参数**推导，可以在设计阶段精确计算；工具调用的内存需求由**并发度、工具实现细节**决定，同样的模型和 batch 配置下，换一个代码沙箱实现（进程 vs 容器 vs 轻量 WASM 沙箱）或换一个检索后端，内存量级可以相差一个数量级——需要**实测压测**而非纯解析估算，只能给出数量级参考。

### 7.5 汇总

| 场景 | CPU 内存峰值量级 | 是否可解析计算 |
|------|-------------------|----------------|
| 无 offload | 十几 GB（可忽略） | 可精确算 |
| 优化器状态 offload（少节点，如 1–2 台） | 数百 GB ~ 1 TB+ | 可精确算（取决于 Φ、节点数） |
| 优化器状态 offload（多节点分片，如 8–16 台） | 数十 ~ 百余 GB / 节点 | 可精确算 |
| Rollout KV cache offload（高并发长序列） | 数百 GB | 可精确算（取决于 S、并发数） |
| Agentic 工具调用（沙箱/检索/浏览器） | 数十 ~ 数百 GB | **需压测**，仅数量级参考 |

> **结论**：不做任何 offload，CPU 内存峰值是十几 GB 量级，可忽略；一旦做优化器状态或 KV cache 的 CPU offload，峰值会跳到**百 GB 到 1 TB+ 量级**，且强烈依赖节点数/并发数；agentic 工具调用则引入了**第三类、且无法纯解析预测**的 CPU 内存开销，取决于工具实现和并发配置，实际部署前应做压测而非只靠公式估算。

## 8. 引入并行配置后：单卡数据传输量、显存与 CPU 内存

前面 §1–§7 的多数数字是**聚合/理想化口径**（如边 A 的 141GB 是「完整权重体量」，边 G 的 423GB/step 已经是单卡口径但隐含训练侧是纯 FSDP、没写明推理侧怎么切）。本节把训练与推理的**具体切分方式（parallelism）**显式引入，重新给出**单卡**视角下的传输量、显存占用与 CPU 内存占用。

### 8.1 典型并行方案假设

70.5B 参数、BF16 权重 141GB，任何单卡（80GB HBM）都放不下，训练和推理都必须切分：

| 阶段 | 切分方式 | 记号 | 典型取值 | 理由 |
|------|---------|------|---------|------|
| 训练（Actor） | FSDP/ZeRO-3，纯数据并行分片（无 TP/PP，沿用 §2 假设） | `N_train` | **64**（8 节点×8 卡） | 多机分片摊薄参数/梯度/优化器状态；不引入 TP 省去激活跨卡通信 |
| 推理（Rollout） | Tensor Parallel | `TP_r` | **8**（单节点内） | GQA 的 8 个 KV head 正好 1 头/卡整除；单序列低延迟生成，避免 PP 的流水线气泡 |
| 推理副本数 | 数据并行（多副本） | `R_r` | 视吞吐需求 | 不影响单卡数字，只影响聚合流量 |

下面所有单卡数字都基于这组配置计算，公式保留 `N_train`、`TP_r` 参数，替换即可重算。边 B/B'/C/D/E 是调度器/缓冲区层面的逻辑量（每条轨迹产出一份，不随内部切分重复），不需要按卡拆分，本节不重复处理。

### 8.2 训练侧单卡显存占用（FSDP 分片）

FSDP/ZeRO-3 把参数、梯度、优化器状态均匀分片到 `N_train` 张卡：

```
参数分片（BF16）  = Φ×2B / N_train  = 141GB / 64  ≈ 2.20 GB/卡
梯度分片（BF16）  = Φ×2B / N_train  ≈ 2.20 GB/卡     （若梯度用 FP32，则 4.41 GB/卡）
优化器状态分片    = Φ×12B / N_train = 846GB / 64 ≈ 13.22 GB/卡  （FP32主权重+一阶矩+二阶矩）
──────────────────────────────────────────────
静态显存合计（不含激活）≈ 17.6 GB/卡（梯度BF16）～ 19.8 GB/卡（梯度FP32）
```

激活内存另计，强依赖 micro-batch 大小、序列长度、是否做 activation checkpointing，量级通常在**十几到几十 GB/卡**（16k 序列、开重计算时），需结合具体训练框架实测，本文不做解析展开。在 80GB 卡上，静态部分只占 ~20GB，给激活留的余量很充足——这也是纯 FSDP（不引入 TP）在这个规模下可行的原因。

### 8.3 推理侧单卡显存占用（TP 分片）

TP 把权重矩阵按行/列切成 `TP_r` 份，KV cache 按 KV head 切分（GQA 8 个 KV head，`TP_r=8` 时正好 1 head/卡）：

```
权重分片（BF16） = Φ×2B / TP_r = 141GB / 8  ≈ 17.6 GB/卡
权重分片（FP8）  = Φ×1B / TP_r ≈ 70GB / 8   ≈ 8.75 GB/卡

单条序列 KV cache（S=16k，全量，见 §7.3）= 5.37 GB
单卡 KV cache（TP 切分后）              = 5.37GB / 8 ≈ 0.671 GB/条

并发 C=64 条时：
单卡 KV cache 合计 = 0.671GB × 64 ≈ 42.9 GB/卡
单卡总占用（权重+KV，不含临时激活缓冲）≈ 17.6 + 42.9 ≈ 60.5 GB/卡（BF16 权重）
```

80GB 卡上还剩 ~20GB 给临时激活/调度缓冲，比较紧张——这正是 §7.3 里「高并发长序列常做 CPU KV cache offload」的直接动因。若把 `TP_r` 降到 4（2 KV head/卡），同样 64 并发时单卡 KV cache 会翻倍到 ~85.9GB，加上权重分片 35.25GB，合计 121GB，**单卡直接放不下**——说明 `TP_r=8` 不是随意选的，是这组（S、并发数）参数下的必要选择。

### 8.4 边 A 权重同步：单卡与聚合流量

§4 的 141GB 是「完整权重体量」，没说清楚在有 TP 切分的推理侧，每张卡实际收多少、网络上总共跑多少——这两个数字不一样：

| 同步方式 | 单卡收到的数据量 | 聚合网络流量（1 个推理副本，`TP_r=8` 张卡） |
|---------|-----------------|---------------------------------------------|
| **朴素广播**（每张卡都发一份完整权重，训练侧不做分片感知） | 141 GB/卡 | `TP_r × 141GB = 1,128 GB`（8× 浪费） |
| **分片感知重分片**（训练侧按推理侧的 TP 切分方式直接发对应分片） | `141GB/TP_r ≈ 17.6 GB/卡` | `141 GB`（等于权重体量本身，无浪费） |

- 分片感知的前提：训练侧 FSDP 的参数切分（按参数展平顺序切）和推理侧 TP 的切分（按矩阵行/列切）**粒度不同**，需要先在训练侧还原出完整参数（这一步本来就是 FSDP 前向要做的 all-gather，不算额外开销，见边 G），再按推理侧需要的维度重新切片分发——工程上可行（veRL 等框架的 colocated 权重同步就是这么做的），但需要专门实现，不是「零成本自动获得」。
- 若有 `R_r` 个推理副本，聚合流量线性扩到 `R_r×141GB`（每个副本都要一份完整权重）；但**单卡收到的数据量与 `R_r` 无关**，仍是 `141GB/TP_r`——这和边 G「单卡通信量与 GPU 数无关」是同一类现象（广播树/环形算法下，单点收发量只取决于该点在拓扑中的位置，不取决于总参与方数量）。

### 8.5 边 G 训练内部通信：单卡口径复核

§4 边 G 的 `423 GB/卡/step` 已经是单卡口径，且在纯 FSDP（`TP_train=1`，即本节 §8.1 的训练假设）下**与 `N_train` 无关**——64 卡还是 128 卡，单卡通信量不变，只是总模型被分成的份数不同。这一点不需要修正。

**需要补充的边界条件**：如果训练侧也引入 TP（不少实际 70B 训练部署会用 TP + FSDP/DP 混合，理由通常是单卡激活显存压力），则每层前向/反向还需要额外的**TP 内激活集合通信**（attention 后、MLP 后各一次 all-reduce，前向反向各算一次），量级 ≈

```
每 step 每卡（TP 激活通信） ≈ 4 × L × (本卡 micro-batch token 数) × h × 2B
```

这个量随「序列长度 × micro-batch token 数 × 隐藏维」增长，与边 G 的 `3Φ`（只随参数量变化）是**不同的增长维度**——序列越长、每卡吞吐的 token 越多，TP 激活通信占比会越明显。本文主模型假设训练侧不引入 TP（§2/§8.1），因此边 G 的 423GB/step 没有包含这一项；若实际部署用了训练侧 TP 混合并行，需要把这项加回去，量级需按具体 micro-batch/序列长度代入计算，本文不做进一步展开。

### 8.6 CPU 内存：对应到具体并行配置的单卡/单节点数字

§7.2、§7.3 的表格本来就隐含了「8 卡/节点」的假设，现在用本节选定的 `N_train=64`、`TP_r=8` 对上号：

| 来源 | 单节点/单卡数字（本节配置下） | 对应 §7 表格行 |
|------|-------------------------------|---------------|
| 优化器状态 CPU offload | 64 卡 = 8 节点 → **106 GB/节点**（÷8 卡记账 ≈13.25 GB/卡） | §7.2「8 节点」行 |
| Rollout KV cache CPU offload（64 并发） | 总量 344GB 分布在 `TP_r=8` 张卡对应的 host 内存上 → **≈43 GB/卡**（NUMA-local） | §7.3 |
| Agentic 工具调用 | 与并行配置基本无关（取决于工具并发数，不取决于 GPU 切分方式），仍是 §7.4 的量级 | §7.4 |

### 8.7 单卡视角小结

| 项目 | 单卡数字（本节配置：`N_train=64` FSDP，`TP_r=8` TP） |
|------|-------------------------------------------------|
| 训练卡静态显存（参数+梯度+优化器状态分片） | ≈17.6～19.8 GB |
| 推理卡显存（权重分片 + 64并发KV cache） | ≈60.5 GB（BF16权重） |
| 边 A 单卡权重同步流量 | 17.6 GB（分片感知）～141 GB（朴素广播） |
| 边 G 单卡训练内部通信 | 423 GB/step，32 step/迭代 ≈13.5 TB/迭代（与 `N_train` 无关，前提训练侧无 TP） |
| 训练侧 CPU 内存（优化器offload，记账到单卡） | ≈13.25 GB/卡（≈106GB/节点） |
| 推理侧 CPU 内存（KV cache offload，单卡） | ≈43 GB/卡（64并发） |

> **结论**：一旦显式引入切分方式，「单卡」层面的数字会比 §1–§7 里那些聚合/理想化数字小得多、也更贴近真实部署会遇到的显存/带宽约束；但**聚合流量**（边 A 的 141GB×副本数、边 G 的 423GB×`N_train` 张卡）该有多大还是多大——切分只是把同一份工作摊到更多张卡上并行完成，不会让总工作量消失。这也是为什么效率优化（分片感知重分片、TP 度数选择、通信-计算重叠）比「堆卡」更重要：堆卡只能线性摊薄单卡负担，摊不掉聚合带宽/存储的总需求。

## 9. 演进趋势：文本 → 多模态，对边 B/C 的冲击与硬件够不够

前面所有测算都假设 prompt/轨迹是纯文本 token。Agentic 场景一旦引入视觉（截图、图片、视频帧作为工具观测或初始任务输入），边 B（prompt 分发）和边 C（轨迹回传）会显著放大——但放大多少，**取决于多模态编码方式**，不是一个固定倍数。

> **⚠ 适用范围声明：本节的「截图进轨迹」只对应 GUI / computer-use agent 这一支，不是 agentic RL 的通用形态。**
> - **主流 agentic RL（数学 / 代码 / 检索 / SQL / QA）的工具返回是文本**（stdout、检索文档、SQL 结果），**不产生图片轨迹**——这正是 §4 边 B'/C 正文假设的纯文本口径，本节的多模态放大对它们**不适用**。
> - **截图作为观测进入多轮轨迹，只在 GUI / computer-use agent（如 UI-TARS-2、Claude Computer Use、DART-GUI）里才是常态**：每一步的 environment observation 本身就是屏幕截图（严格说是**环境观测=边 B'**，不是 function-calling 意义上的「工具返回」）。下面的算例应理解为「GUI-agent 场景」，而非所有 agentic RL 都要付这笔账。

### 9.1 为什么多模态会放大

**连续视觉特征**（LLaVA / Qwen-VL / GPT-4V 这一类目前主流的做法）：视觉编码器（ViT）输出的是连续向量，**不存在小词表 id 可以还原它**——要在训练侧精确复现同一份输入（GRPO 的 old/ref logprob 依赖 teacher-forcing 复现），只有两个选择：① 缓存**原始图像字节**（压缩后的 JPEG/PNG，训练时重新跑一遍视觉编码器）；② 直接缓存**视觉编码器输出的连续 embedding**（不用重新跑编码器，但每个视觉 token 要按 `h×2B` 存，而不是 4B 的 id）。方案②每个视觉 token 的字节成本是文本 token 的 **2000 倍量级**（`h=8192` 时是 `8192×2B / 4B = 4096` 倍）。

本文之前的边 B/C 公式（`N_p×P×4B`、`B×S×4B`）隐含假设的正是"处处 4B/token"的离散口径——一旦多模态用了方案②，这个假设本身就不成立了。

### 9.2 worked example：加上视觉工具调用后重新测算边 B/C

假设一条 agentic 轨迹里有 `k=10` 次工具调用返回截图（1280×720，对应 §6.1 的"长程 agent"场景），每张图按主流 VLM 惯例编码成 `576` 个视觉 token，投影到本文模型的隐藏维 `h=8192`（BF16）：

```
文本口径（原基线）：每条轨迹边 C 字节数 = S×4B = 16,384×4 = 64 KB

方案①（压缩原始图像字节，≈400KB/张，训练时重新过一遍视觉编码器）：
  每条轨迹新增 = 10 × 400KB = 4 MB
  新边 C 总量  = B×(64KB+4MB) ≈ 16,384×4MB ≈ 64 GB   （相对基线 1.07GB，约 60 倍）

方案②（缓存连续视觉 embedding，576 token/图 × 8192×2B/token）：
  每张图 = 576 × 16KB = 9.0 MB；每条轨迹新增 = 10×9.0MB = 90 MB
  新边 C 总量  = B×90MB ≈ 16,384×90MB ≈ 1.44 TB      （相对基线 1.07GB，约 1350 倍）
```

**方案①是几十倍量级，方案②是超千倍量级**，实际系统用哪种，主要看是否愿意为了省内存/带宽而承担训练侧重新过一遍视觉编码器的算力开销。边 B（初始 prompt 若也带图）按 `N_p=1024` 张同样的图算，方案①≈410MB（约 12 倍于 33.5MB 基线），方案②≈4.6GB（约 140 倍）——比边 C 涨得少，因为 prompt 只发一份、不像轨迹要按 `B=16,384` 份重复算。

> **这两个数是「上界」，不是「预期值」——真实 GUI-agent 训练会主动压低它：**
> 1. **每图 token 数可能更高，也可能被压低。** 576 token/图 取的是 LLaVA-1.5 的保守值；现在 GUI agent 多用 Qwen2-VL / UI-TARS 的**原生分辨率**编码，一张 720p 截图为了让 UI 小字可读常要 **1000~1600 token**，方案②的真实成本可能比 1.44TB 更高。反过来，若刻意降分辨率/做 token 压缩（pixel-shuffle 等），又能砍下来。
> 2. **「10 张图全程满 token 保留」本身就是最坏情况。** 视觉 token 太贵，真实系统会**激进裁剪历史截图**——只保留最近 N 张（keep-last-N），或把旧截图替换成文本描述。所以 1.44TB 是「什么优化都不做」的上界，而这个上界恰恰是工程上被专门规避的对象；叠加 keep-last-N 后实际留存量会**大幅低于**此值。
> 3. GUI 任务实际步数常达 15~50 步（比这里假设的 10 更长），但正因为每步一张图太贵，才更依赖上面的裁剪策略——**步数与裁剪两个因素叠加，才是真实留存量**，本文不展开具体策略下的精确值，只给「不裁剪」的上界作参照。

### 9.3 硬件够不够（一）：华为 Atlas 900 A3 SuperPoD（昇腾 910C / CloudMatrix384）

| 维度 | 规格 | 相对本文新增负载 |
|------|------|------------------|
| 单卡 HBM | **128 GB**（910C，2 die×64GB），带宽 ≈3.2 TB/s | 比 §8 假设的 80GB(Nvidia口径) 更宽松；视觉 embedding 若直接进 KV cache/激活，序列位置数只涨 1.3~2×（576×10÷16384），远没到"几百倍"——那个倍数是**存储/传输侧**的问题，不是显存里序列长度的问题 |
| 超节点 UB 互联（scale-up，卡间） | 单卡 >392 GB/s 单向 | 边 A/G 已经是 TB 级流量的常规负载，方案②新增的 1.44TB/迭代边 C 传输，量级上和边 G（13.5TB/迭代）同一个数量级，UB 带宽富余 |
| Scale-out RDMA（跨超节点，RoCE） | 单卡 400 Gbps ≈ 50 GB/s 单向 | 以 1 节点 8 卡聚合 ≈400GB/s 估算，搬 1.44TB 只需 ≈3.6s，相对一次 RL 迭代通常的分钟级耗时**不是瓶颈** |
| 超节点总 HBM | 384×128GB = 48 TB | 富余，多模态新增负载不会顶到这个量级 |

**结论**：网络互联（UB scale-up + RoCE scale-out）和单卡 HBM 在这组新增负载下都**明显够用**，不是瓶颈——即使是方案②的 1.44TB/迭代，也只是把边 C 从"可忽略"变成"和边 G 同量级"，没有超出这套硬件本来就要扛的通信规模。

> 目前华为官方公开的是 **Atlas 900 A3 SuperPoD**（昇腾 910C，已量产/部署），下一代披露的是 **Atlas 950 SuperPoD**（昇腾 950DT，144GB HBM/卡、4TB/s 带宽，规划 2026 Q4）。

### 9.4 硬件够不够（二）：CPU 内存，现在 vs 未来 1.5 TB/节点

真正吃紧的是 **CPU（host）内存**——多模态经验缓冲区（边 C 新增的 64GB～1.44TB）如果都堆在同一批节点的 host RAM 里，会和 §7/§8 已经算过的其他 CPU 内存需求叠加：

| 场景 | 多模态经验缓冲区 | + 优化器状态 offload（§8.6, 106GB/节点） | + KV cache offload（§8.6, ≈344GB/8卡节点） | 合计（若都堆在同一节点） | 对比 1.5TB |
|------|-----------------|----------------------------------------------|----------------------------------------------|--------------------------|-----------|
| 方案①（压缩字节，60×） | 64 GB（若集中不分片） | 106 GB | 344 GB | ≈514 GB | 34%，**明显够用** |
| 方案②（连续embedding，1350×），**集中式**（不分片） | 1.44 TB | 106 GB | 344 GB | ≈1.89 TB | **超出 1.5TB**，不够 |
| 方案②，**按 8 节点分片**（同 §8 的 `N_train=64`/8节点配置） | 1.44TB÷8 ≈180 GB/节点 | 106 GB | 344 GB（如与rollout同节点） | ≈630 GB | 42%，**够用** |

**关键结论不是一句"够"或"不够"，而是取决于两件事：**
1. **多模态编码方案**——方案①（压缩字节+训练侧重算）现在的内存就够，未来 1.5TB 更是绰绰有余；方案②（缓存连续 embedding）不分片的话，即使未来 1.5TB/节点也**扛不住**。
2. **经验缓冲区是否分片**——只要把多模态经验数据像 §8 的优化器状态/KV cache 一样，**按训练/rollout 节点数分片存放**而不是集中在一两台机器上，即便是最激进的方案②，每节点份额也能压到 1.5TB 预算之内的一半左右，留有余量应对 tool 调用（§7.4）等其他 CPU 内存开销。

工程上更稳妥的做法是**优先选方案①**（存压缩原始图像/视频帧，训练时重新过一遍视觉编码器）：CPU 内存和网络代价都低一个数量级，只是多花一点训练侧算力去重新编码——这笔算力开销远小于它换来的内存/带宽节省，本文不做进一步展开。

### 9.5 对照：当前最主流的 coding agent 场景（纯文本，但沿另一个轴放大）

Coding（尤其 SWE-agent：在真实仓库里读文件、改代码、跑测试）是目前 agent 落地最广的场景，但它和上面的 GUI/多模态是**两种完全不同的放大方式**——理清这个对照，比单看多模态更贴近现实。

| 维度 | GUI/多模态（§9.1–9.4） | Coding agent（本节） |
|------|--------------------------|----------------------|
| 工具/环境返回 | 截图（图像） | **纯文本**：stdout/stderr、文件内容、测试 traceback、grep 结果 |
| 放大轴 | bytes-per-token（连续 embedding，×2000/token） | **序列长度 S**（token 数变多，仍是 4B/token） |
| 边 C 字节率 | 可能爆到 `h×2B`/token | **不变，仍 4B/token id**（回到 §6.1 的口径） |
| 主要瓶颈 | 显存/传输侧的图像数据 | **沙箱 CPU 内存 + 奖励执行墙钟时间** |

**1. 边 C/E：沿 S 放大，不沿 bytes-per-token 放大。**
Coding 的观测是文本，所以边 C 仍按 4B/token id 走——多模态那套 ×60~×1350 的放大**完全不适用**。放大来自 S 变长：SWE 轨迹常有 20~50 轮（读文件→改→跑测试→看 traceback→再改），叠加单次工具返回可能是整个文件/长 traceback，S 很容易从 16k 涨到 32k~128k。直接套 §6.1，无需新公式：

```
S=64k（P8k + R56k，约对应长程 SWE 任务）：
  边 C = B×S×4B = 16,384×65,536×4 ≈ 4.29 GB   （即 §6.1 的 64k 行）
```

**2. response_mask 让边 E 亚线性增长——coding 特有的省法。**
Tool-heavy 轨迹里，R 区大部分是**观测 token**（文件内容、测试输出），只有小部分是模型生成的**动作 token**。而 old/ref logprob 只在动作 token 上有意义（观测 token 的 `response_mask=0`，被排除出 GRPO loss，见 §1 边 C 的 mask 说明）。所以边 E 的**存储量按「生成 token 数」而非「全部 R」算**：

```
设生成 token 仅占 R 的 ~25%（观测占大头）：
  生成 token 数 = B×R×0.25 = 16,384×57,344×0.25 ≈ 234.9M
  边 E(old) = 234.9M×4B ≈ 0.94 GB   （ref 同）
```
对比 S=16k 全生成基线的 536.9MB：R 涨了 7×，边 E 只涨 ~1.75×——多出来的长度几乎都是不需要 logprob 的观测。（这也说明 §6.1 里 64k 行的 3.76GB 是「R 全部生成」的上界；coding 里 `response_mask` 把它砍到 ~0.94GB。）**但前向算力省不掉**：teacher-forcing 前向必须把全部 S 个 token 都作为上下文过一遍（注意力需要它们），所以 §4 边 G 的「额外前向 4.5 TB/GPU」和注意力 O(S²) 的算力**照样按完整 S 付**。一句话：coding 里观测 token **省 logprob 存储、不省前向算力**。

**3. 真正的瓶颈搬到了沙箱 CPU 内存 + 奖励墙钟——不是数据传输。**

- **沙箱**：coding 的每条 rollout 轨迹几乎全程需要一个活的执行环境（checkout 好仓库、装好依赖的容器）。这直接落到 §7.4，且量级远超那里的通用估计：SWE 类容器常驻 RSS ~0.5~2 GB/个，预构建镜像 1~5 GB（磁盘）。设并发 512 条：

  ```
  沙箱 CPU 内存 ≈ 512 × 2GB ≈ 1 TB    （比 §7.4 的通用示例 96GB 高一个数量级）
  镜像磁盘占用：跨多仓库 RL 时可达数百 GB ~ TB 级
  ```
  这再次印证 §7.4 的定性——沙箱内存**不可解析、需压测**，且在 coding 场景是**头号 CPU 内存来源**，甚至超过优化器 offload。

- **奖励墙钟**：SWE 的奖励通常是「跑测试套件是否通过」的稀疏标量。传输数据仍可忽略（边 D ~128KB 不变），但**跑一遍测试要秒级~分钟级**，成为 rollout 的墙钟瓶颈。这正是 coding agentic RL 强依赖**异步 / partial rollout**（§6.5 off-policy 容忍、§10 TransferQueue 全异步）的根本原因——不是为了省带宽，是为了让慢速奖励不阻塞训练。

**小结**：对 coding 场景，跨阶段**字节账**几乎就是 §6.1「更长 S」的直接代入（4.29GB@64k），没有多模态那种质变；真正需要重新算的是 **§7.4 的沙箱 CPU 内存（~TB 级，成头号来源）** 和**奖励执行的墙钟时间**（催生异步 RL）。一句话：**coding agent 的 scaling 压力在「沙箱 + 慢奖励」，不在数据传输本身。**

## 10. 工程实现映射：经验缓冲区在 verl 里的落地（TransferQueue）

前面 §1–§9 把经验缓冲区、边 C/D/E/F 都当成抽象的「阶段」和「边」来测算。在真实框架里，这套「rollout 产数据 → 缓冲区暂存 → 训练侧取用」的数据流有一个具体的开源实现：华为昇腾团队开源、并于 **2026 年 4 月正式合入 verl** 的 **TransferQueue**——一个面向 RL 后训练的**异步流式数据管理模块**。它在 128×H100 的多模态后训练上带来约 **49% 的端到端提速**，恰好印证了 §9 关于「多模态放大后，单点数据通路会成为瓶颈」的判断。

### 10.1 它对应本文的哪些部分

| 本文抽象概念 | TransferQueue 里的对应物 |
|--------------|--------------------------|
| 经验缓冲区（Experience Buffer） | TransferQueue 本身（分布式存储 + 中转） |
| 边 C/D/E「读取后写回缓冲区」 | 各生产者按**字段**写入：rollout 写 token、验证器写 reward、训练侧写 old/ref logprob |
| 边 F「mini-batch 取数」 | 下游任务按需请求，控制面确认字段齐全后才触发数据面搬运 |

### 10.2 关键设计（与本文测算的呼应）

1. **控制面 / 数据面分离**——对应 §4「边 F 传的是轻量 token 流 + 少量浮点数组」。它把 verl 原来那个所有数据都要过一遍 `RayPPOTrainer`（单点瓶颈）的 `DataProto`，拆成 **BatchMeta**（元数据，轻，由 `TransferQueueController` 管理）+ **TensorDict**（实际张量，重，由 `StorageManager` 存）。取数时先比对元数据「字段是否都就绪」，再决定是否真的搬运张量。

2. **字段级就绪 + 独立消费**——对应边 C/D/E 被画成「从缓冲区读、算完写回」的后处理结构。它按字段追踪每条样本的生产/消费状态：**所有必需字段（token、reward、old/ref logprob）都 ready 后**，该样本才可被下游消费；且打分、算 logprob、优化器 step 各自维护独立的消费历史，互不干扰地读同一份数据。

3. **通信优化：单 rank 代理 + 组内 broadcast**——对应 §8 讨论的「集中式 vs 分片」和边 F 的单卡代理取数。在每个数据并行组里只指定**一个 rank** 与 TransferQueue 交互，取回后再 broadcast 给组内其他 rank，避免大规模场景下每卡单独请求造成的请求风暴。

4. **rollout 与 trainer 完全解耦 → 全异步**——数据持续流经 TransferQueue、参数更新走 CheckpointEngine，把原本串行的「生成→训练」混合负载变成可并行的流水线。需注意：verl 的 TransferQueue 默认 **FIFO、偏向流式处理最新数据**，因此更适配 **PPO/GRPO 这类 on-policy（或近 on-policy）算法**，与本文假设的 GRPO on-policy 场景一致。

### 10.3 与多模态、CPU 内存的关系

TransferQueue 那 49% 提速的 benchmark **正是多模态后训练**——这与 §9 的结论直接呼应：一旦边 C 从 1.07GB 放大到 GB～TB 级（§9.2），传统「所有数据过单点 trainer」的 `DataProto` 方式就撑不住了，而 TransferQueue 的**分布式存储**天然就是 §9.4 强调的「经验缓冲区按节点分片」在框架层的落地——这也是为什么它能把多模态那种放大后的数据流扛住。社区还有把 **openYuanrong 作为 TransferQueue 的 KV 后端**进一步加速 veRL 样本传输的工作。

### 10.4 样本数据落在哪里：存储后端决定内存分层

一个常见误解是「开了 TransferQueue，样本数据就不用搬到 host 内存了」。实际上 TransferQueue 是**数据管理层**，不是固定的物理存储位置——样本张量（TensorDict）到底躺在 HBM、host DRAM 还是 SSD，由**可插拔的 StorageManager 后端**决定。它真正消除的是「所有数据挤过单个 trainer 进程」的**单点瓶颈**，而不是**内存占用/host 搬运本身**：

| 存储后端 | 数据物理位置 | 是否需搬到 host | 对应本文哪一节 |
|---------|-------------|----------------|---------------|
| **SimpleStorage**（默认，`SimpleStorageUnit`） | **host CPU DRAM**，按节点分布式存放 | **需要**：rollout 在 HBM 产出后做 device→host 拷贝 | §7.1 的 host 经验缓冲区 + §9.4 的「按节点分片」 |
| **Yuanrong**（昇腾原生分层存储） | 分层 **HBM / DRAM / SSD** | 热数据可留 HBM，不强制落 host | §8 的 HBM 富余量决定能留多少 |
| **MooncakeStore** | KV 存储，**GPU↔DRAM 走 RDMA** | RDMA 直传，绕开传统 host staging | — |
| **RayRDT** | Ray actor 间直接传对象 | 依赖 Ray 对象存储放置策略 | — |

回扣前面几节的三点：

1. **默认后端仍然搬到 host**。SimpleStorage 就是把样本存进 host DRAM——只不过是**按节点分布式**（呼应 §9.4「经验缓冲区按节点分片」），而非老 `DataProto` 那样全 funnel 到一个进程。所以默认配置下，§7.1（纯文本 ~2.15GB）和 §9.4（多模态 GB～TB）算的那些 host 内存占用**依然成立**，TransferQueue 省的是单点通路，不是这块内存。

2. **想免 host 搬运得显式换后端**。Yuanrong 的 HBM 层或 Mooncake 的 RDMA GPU↔DRAM 直传，才能让热数据留在 HBM / 走 RDMA 绕开 host——这正是社区把 openYuanrong 接成 KV 后端的动机。

3. **但 HBM 容量把大数据逼回 host/SSD**。回到 §8 的账：推理卡 80GB HBM 已被权重分片（17.6GB）+ KV cache（~43GB）占到 ~60GB，训练卡也被参数/梯度/优化器分片 + 激活占着。纯文本经验数据才 2.15GB 尚可留 HBM，但 §9.2 方案②的 **1.44TB 根本不可能全留 HBM**——这种量级只能落 host DRAM 或 SSD。**「是否搬到 host」本质上是被 HBM 容量逼出来的分层决策**：小数据可留 HBM，大数据（尤其多模态连续 embedding）绕不开 host/SSD。

> 参考：verl 官方文档 *TransferQueue Data System*、`Ascend/TransferQueue`（GitHub）、openEuler 关于 openYuanrong 作为 TransferQueue KV 后端的博客。

## 11. 关键洞察

> 综合 §1–§10 的全部测算，按「数据平面 → 计算通信 → 内存 → 场景 → 系统实现」五层归纳。

**数据平面**

1. **跨阶段传输几乎全是权重同步。** 经验数据（C+D+E，token id + 标量 logprob + 奖励）约 **2.15 GB**，权重同步 **141 GB**，占 143.2 GB 跨阶段总量的 **98.5%**。权重同步是常数项（只随 Φ、精度、同步频率变），经验数据随 `N_p×G` 与 `S` 线性增长——两者随规模拉大而分化（§6.1），但绝对值上权重同步长期仍是跨阶段头号项。**优化跨阶段带宽 ≈ 优化权重同步**：FP8 减半（§6.4）、off-policy 摊薄（§6.5）、分片感知重分片避免朴素广播的 8× 浪费（§8.4）。

2. **经验数据小，是因为只传 id 不传分布。** 若传每 token 完整 logits 会是 68.9 TB（现方案的 **6.4 万倍**）。这条「传 id 不传分布」红线是 RL 框架能跑通的前提——也正是多模态方案②（缓存连续 embedding）打破它、把边 C 推到 TB 级的根源（§9.2）。

**计算与通信**

3. **训练内部通信（边 G）才是绝对大头。** ~13.5 TB/GPU/迭代，比整个跨阶段数据平面高约 **94×**。它只取决于 `n_steps = N_p/mini` 与 Φ，**与 group size G、序列长度 S、batch 大小都无关**（§6.2）——增大 G 只增加每步 FLOPs 与经验数据量，不增加 optimizer 通信次数；减小 mini 才会线性抬高边 G（§6.3 的显存↔通信权衡）。

4. **GRPO 参考模型是隐藏成本。** KL 项要求参考模型全量前向 **4.5 TB/GPU**，与 old-logprob 前向同量级、与优化器 step 同数量级。其中 old-logprob 前向可复用 rollout 生成期 logprob 省掉（有数值口径代价），但 ref 前向刚性、省不掉（§12）。

**内存**

5. **显存靠分片摊薄，但聚合工作量不消失。** 引入并行（64 卡 FSDP + 推理 TP=8）后，训练卡静态显存降到 ~18 GB、推理卡 ~60 GB（§8）；但边 A 的 141GB×副本、边 G 的 423GB×卡数这些**聚合量**不变——切分只把工作摊到更多卡并行，摊不掉总需求。TP 度数也不是随意选的：TP=8 是被 KV cache 显存逼出来的（TP=4 时单卡放不下，§8.3）。

6. **CPU 内存是被低估的硬约束，且有三类来源。** 不做 offload 仅十几 GB；优化器状态 / KV cache offload 后跳到百 GB~1TB+（§7.2/§7.3，可解析）；agentic 工具调用（尤其 coding 沙箱）是第三类、不可解析、需压测的来源（§7.4）。三者叠加才决定 host RAM（乃至未来 1.5TB/节点）够不够（§9.4）。

**场景与演进**

7. **Agentic 放大有两个正交的轴，别混为一谈。**
   - **沿 `S` 放大（主流：coding / 长程工具链）**：观测是文本，边 C 仍 4B/token，只是 S 变长（§6.1）；`response_mask` 让边 E 亚线性增长（观测 token 不算 logprob），但前向算力仍按完整 S 付；真正瓶颈搬到**沙箱 CPU（~TB）+ 奖励墙钟**，催生异步 RL（§9.5）。
   - **沿 bytes-per-token 放大（GUI / 多模态）**：截图进轨迹，连续 embedding 每 token 贵 ~2000×，边 C 最坏到 1.44 TB（§9.2）——但这是 GUI-agent 专属、且是「不裁剪」上界，真实用 keep-last-N 规避（§9.1–9.4）。

**系统实现**

8. **数据通路的瓶颈是「单点」，不是「字节」。** 经验数据本身才 GB 级，但老式全部 funnel 到单个 trainer 进程（DataProto）在多模态放大后撑不住；TransferQueue 用控制面/数据面分离 + 分布式存储解决（§10），其存储后端还决定数据落 HBM/DRAM/SSD——但默认落 host DRAM，并不会让内存占用消失（§10.4）。

## 12. 瓶颈与优化建议

按五类归纳（括号为对应小节），从跨阶段数据平面一路到数据管理：

| 类别 | 维度 | 问题 | 优化 |
|------|------|------|------|
| **跨阶段** | 权重同步（A） | 分离部署 141 GB/步跨集群 | colocated 共卡 + CUDA IPC；FP8 同步（§6.4）；off-policy 降频摊薄（§6.5）；分片感知重分片避免 8× 广播（§8.4）；bucket 化 overlap |
| | 经验数据（C/E） | 随 `N_p×G×S` 增长 | 只传 token id 不传分布；`response_mask` 免存观测 token 的 logprob（§9.5） |
| **训练计算** | 训练通信（G） | ~13.5 TB/GPU/迭代 | TP+PP 混合并行、序列并行、通信-计算重叠、序列打包去 padding、梯度 BF16 reduce（§8.5） |
| | 参考模型（KL） | 4.5 TB/GPU 前向（刚性） | 参考 logprob 缓存 / 低频刷新；降低 KL 刷新频率；固定间隔重算（★省不掉，不像 old logprob 可复用生成期数据） |
| | old logprob 前向 | 4.5 TB/GPU 前向 | 复用 rollout 生成期 logprob 跳过重算；代价是推理/训练引擎数值路径不同带来的 ratio 噪声，需结合 clip range 权衡（见 §4 边 E） |
| | Rollout 生成 | 长序列 + 长尾轨迹拖批 | 连续批处理、partial rollout、异步 off-policy、长度惩罚 |
| **内存** | 训练显存 | 优化器状态 ~846 GB + 激活 | FSDP/ZeRO-3 分片（§8.2）、CPU offload、优化器 8-bit |
| | 推理显存 | 权重 + KV 占满 80GB（§8.3） | TP 度数选择、KV cache CPU offload（§7.3）、FP8 权重 |
| | CPU 内存 | offload 后百 GB~1TB（§7.2） | 按节点分片、大内存机型；工具沙箱不可解析、需压测（§7.4） |
| **场景** | 多模态数据 | 边 C→TB 级（§9.2） | 优先方案①（压缩字节 + 训练侧重算）；keep-last-N 裁剪历史图；经验缓冲区按节点分片（§9.4） |
| | Coding agent | 沙箱 ~TB + 奖励墙钟（§9.5） | 沙箱池化 / 复用、镜像分层缓存；异步 / partial rollout 让慢奖励不阻塞训练 |
| **数据管理** | 经验缓冲区 | 单点 funnel 瓶颈（§10） | TransferQueue 控制面/数据面分离、分布式分片存储、rollout↔train 全异步解耦；选 HBM/RDMA 后端省 host 搬运（§10.4） |

## 13. 总结

以 Llama-3-70B 的 agentic GRPO 后训练为例（1024 prompt × G=16 = 16,384 轨迹/迭代，mini=32 prompt → 32 optimizer step）：

**数据传输量的三个数量级**

- **跨阶段传输 ≈ 143.2 GB/迭代，其中 98.5% 是权重同步**；真正的经验数据仅 ~2.15 GB——因为只传 token id 而非分布（否则会是 68.9 TB）。
- **训练内部集合通信 ~13.5 TB/GPU/迭代，高出跨阶段约 94×，才是吞吐的真正瓶颈**，再叠加 GRPO 参考模型 4.5 TB/GPU 的隐藏前向。
- 以上都是**聚合口径**；引入并行切分（64 卡 FSDP + 推理 TP=8）后单卡显存/流量小得多、更贴近真实约束，但聚合工作量不因切分消失（§8）。

**真实部署的紧约束是 CPU 内存**

- 不做 offload 仅十几 GB；一旦优化器状态 / KV cache offload，跳到百 GB~1TB+。
- agentic 工具（尤其 coding 沙箱）是**不可解析、需压测**的第三类来源（§7、§9.5）。

**演进沿两条正交的轴**

- **主流的 coding / 长程工具链——沿 `S` 放大**：数据账仍是文本，瓶颈搬到沙箱与奖励墙钟，催生异步 RL。
- **GUI / 多模态——沿 bytes-per-token 放大**：边 C 最坏到 1.44 TB，但可用「压缩字节 + 训练侧重算」「keep-last-N」「经验缓冲区按节点分片」压下来（§9）。

**工程实现**

- 经验数据通路的瓶颈从来是「单点」而非「字节」——TransferQueue 的控制面/数据面分离 + 分布式分片存储正是答案（§10）。

---

> **归结为三句：** 跨阶段带宽 = 优化权重同步；训练吞吐 = 优化边 G 与并行策略；真实部署的墙 = CPU 内存与（coding 的）沙箱 + 慢奖励。
