# Agentic RL 后训练：各阶段间数据传输量测算

> 模型：**Llama-3-70B**（80 层，hidden = 8192）  ·  算法：**GRPO**
> 初始请求长度 8k  ·  global batch（**prompt 数**）= 1024  ·  mini batch = 32（**prompt 数**，与 global batch 同单位，惯例做法）

本文逐条测算 agentic 强化学习后训练流水线中**跨阶段（stage 之间）** 的数据传输量，给出公式、代入值与可复算结果，并区分「跨阶段数据平面」与「训练内部集合通信」两类流量。所有假设都参数化，替换即可重算。

---

## 1. 流水线架构与阶段划分

GRPO 后训练系统由**推理引擎（Rollout）**、**奖励/验证器**、**训练引擎（Actor 更新）**、**参考模型（KL 项）** 构成。整张图是**一次 RL 迭代（outer loop，执行 1 次）**；训练引擎内部嵌套了一个 **optimizer step 内层循环（inner loop，执行 32 次）**——(G) 的 13.5 TB/GPU 就是这 32 次单 step 通信量（每次 423 GB/GPU）累加出来的。每条边 (A)–(G) 标注方向与单次触发的传输量。

```
        ┌───────────────────────┐
        │   调度器 / 数据集       │
        └───────────────────────┘
                    │  (B) prompt 分发            33.5 MB
                    ▼
        ┌───────────────────────┐  (B') 环境/工具观测    ┌──────────────┐
        │      Rollout 引擎      │ ───────────────────▶ │  环境 / 工具  │
        │    (vLLM / SGLang)    │ ◀─────────────────── │ (code/search)│
        │  每 prompt 采样 G=16 条 │   (拼回上下文, 抬高 S) └──────────────┘
        └───────────────────────┘ ◀───────────────────────────────┐
                    │  (C) 轨迹 token 回传   1.07 GB               │
                    ▼                                             │
        ┌───────────────────────┐                                 │
        │     奖励模型 / 验证器   │                                 │
        └───────────────────────┘                                 │
                    │  (D) 奖励 + 优势   ~128 KB                   │  (A) 权重同步（迭代末尾，×1）
                    ▼                                             │  Φ×2B = 141 GB
        ┌───────────────────────┐ ◀── (E) 旧 logprob 536.9 MB     │  (回环: 训练 → Rollout)
        │     经验缓冲区          │ ◀── (E) 参考 logprob 536.9 MB   │
        │   (Experience Buffer) │      (GRPO KL 项)               │
        └───────────────────────┘                                 │
                    │  (F) 切成 32 个 mini-batch                   │
                    │      (每个 = 32 prompt × G=16 = 512 条轨迹)   │
                    ▼                                             │
   ┌──────────────────────────────────────────────────────┐       │
   │   optimizer step 内层循环 (inner loop, ×32)           │       │
   │                                                      │       │
   │   取 1 个 mini-batch（512 条轨迹，本地 ≈33.5 MB）       │       │
   │                │                                     │       │
   │                ▼                                     │       │
   │   ┌───────────────────────┐                          │       │
   │   │    训练引擎 (Actor)      │ ◀── (G) 单 step 通信    │───────┘
   │   │    FSDP / ZeRO-3       │      423 GB/GPU         │          
   │   └───────────────────────┘   (all-gather×2 +        │          
   │                │                reduce-scatter×1)    │          
   │                └──── 更新参数分片, 回到下一个 mini-batch │          
   │                       （循环 32 次后退出）              │          
   └──────────────────────────────────────────────────────┘ 
        32 步累计 (G) = 32 × 423 GB/GPU ≈ 13.5 TB/GPU
```

**阶段（stage）与阶段间的「边（edge）」：**

| 边 | 方向 | 传输内容 |
|----|------|----------|
| A | 训练引擎 → Rollout 引擎 | **模型权重同步** |
| B | 调度器 → Rollout 引擎 | 采样用的 prompt token |
| B' | 环境/工具 ↔ Rollout 引擎 | 多轮 agentic 的观测（工具返回、环境反馈） |
| C | Rollout 引擎 → 经验缓冲区 | 生成轨迹的 token id、mask |
| D | 奖励模型/验证器 → 缓冲区 | 标量奖励、group 归一化后的优势 |
| E | logprob worker → 缓冲区 | 旧策略 logprob + 参考策略 logprob（GRPO KL 项） |
| F | 缓冲区 → 训练引擎 | mini-batch 取数（本地/同机，通常不跨网） |
| G | 训练引擎内部 | 前向/反向的参数 all-gather 与梯度 reduce-scatter |

## 3. 模型规模拆解：为什么权重同步是 141 GB

Llama-3-70B 单层参数（GQA：64 Q head / 8 KV head，SwiGLU，`i`=28672）：

| 组件 | 计算 | 参数量 |
|------|------|--------|
| q_proj | 8192×8192 | 67.1 M |
| k_proj / v_proj | 8192×1024 ×2 | 16.8 M |
| o_proj | 8192×8192 | 67.1 M |
| gate / up / down | 8192×28672 ×3 | 704.6 M |
| **单层合计** | | **855.6 M** |
| ×80 层 | | 68.4 B |
| embedding + lm_head（未 tie） | 128256×8192 ×2 | 2.1 B |
| **总计 `Φ`** | | **≈ 70.5 B** |

- **BF16 权重体量** = `70.5e9 × 2 B` = **141 GB** → 这是每次权重同步（边 A）的基础量，**与 batch 大小无关**。
- FP32 主权重 282 GB、Adam 一/二阶矩 2×282 GB、梯度 141–282 GB —— 这些是**显存占用**，常驻训练侧、不跨阶段传输。

## 4. 逐边数据传输量

### 边 A — 权重同步（训练 → Rollout）★ 主导项

每次迭代（on-policy）需把更新后的 Actor 权重刷进推理引擎：

```
V_A = Φ × bytes_param = 70.5e9 × 2 = 141 GB / 迭代
```

- **FP8 同步**：≈ **70 GB**（推理引擎支持 FP8 权重时）。
- **拓扑影响**：
  - *同机 colocated*（veRL 等，训练与推理共用 GPU）：走 NCCL broadcast / CUDA IPC，本地 NVLink，141 GB 在 ~数百 GB/s 下 ≈ 0.3–1 s。
  - *分离部署 disaggregated*（训练集群 ↔ 推理集群）：141 GB 每步跨集群 IB/以太网，是**头号瓶颈**。
- **频率**：严格 on-policy 每迭代 1 次；若允许 off-policy（每 N 步同步一次）则摊薄为 `141/N`。

### 边 B — Prompt 分发（调度器 → Rollout）

只需把 **1024 个 prompt 各发一份**，由 Rollout 引擎在采样时对每条 prompt 各生成 `G=16` 条轨迹（无需重复发送 prompt 本身）：

```
V_B = N_p × P × 4 B = 1024 × 8192 × 4 = 33,554,432 B ≈ 33.5 MB
```
可忽略。

### 边 B' — Agentic 多轮环境/工具交互

多轮 rollout 中，工具返回（代码执行结果、检索文档、环境状态）会**拼回上下文**。字节量本身不大（文本），但**显著抬高 `S`**，从而放大边 C/E 的传输与训练算力。设平均 `k` 轮、每轮观测 `o` token：

```
S = P + Σ(生成_i + 观测_i)   →  R 随轮数线性增长
```
这是 agentic 场景相较普通 RLHF 的关键放大因子（见 §8 敏感性）。

### 边 C — 轨迹回传（Rollout → 缓冲区）

只传 **token id + attention mask**，不传激活/logits。轨迹总数 `B = N_p × G = 16,384`：

```
总 token 数 = B × S = 16,384 × 16,384 = 268,435,456 ≈ 268.4 M tokens
V_C = 268.4M × 4 B (int32 id) = 1,073,741,824 B ≈ 1.07 GB   (+ mask 约同量级或更小)
```

### 边 D — 奖励与优势（验证器/RM → 缓冲区）

GRPO 在数学/代码任务多用**规则验证器**，奖励是每条轨迹一个标量；优势在**组内**（同一 prompt 的 G=16 条轨迹）做均值/方差归一化后广播回每条轨迹：

```
V_reward = B × 4 B = 16,384 × 4 = 65,536 B = 64 KB
V_adv    = B × 4 B = 64 KB          （组内归一化优势，按序列广播）
总计 ≈ 128 KB
```
可忽略。若用 **神经奖励模型（另一个 70B）**：输入即边 C 的 token（同量级），输出标量极小；但 RM 前向是**算力**开销，激活不跨阶段。

### 边 E — 旧策略 / 参考策略 logprob 回传

只在**响应 token**上计算每 token 的选中 token 对数概率（FP32）：

```
响应 token 数 = B × R = 16,384 × 8192 = 134,217,728 ≈ 134.2 M
V_old_logp = 134.2M × 4 B = 536,870,912 B ≈ 536.9 MB   （π_old，用于重要性比值）
V_ref_logp = 134.2M × 4 B ≈ 536.9 MB                   （π_ref，GRPO 的 KL 惩罚项）
```

GRPO 需要**参考模型的额外一次全量前向**（算力，见边 G 附加项）+ 上述 536.9 MB 参考 logprob 跨阶段传输。

> **`π_old` 前向是否必须？—— 一个可选但有代价的优化。**
> Rollout 引擎生成时为了采样，本身就会算出每个生成 token 的 logits/logprob，理论上可以直接复用这份 `log π_old`，省掉训练引擎侧那次专门的全量前向（§4 边 G 附加项里的 4.5 TB/GPU）。但多数生产级框架（veRL、OpenRLHF 等）**默认仍然重算**，原因是：
> 1. 推理引擎（vLLM/SGLang，常用 PagedAttention + 低精度/FP8）与训练引擎（FSDP，BF16/FP32）数值路径不同，同一份权重算出的 logits 存在细微漂移；
> 2. GRPO/PPO 的 `ratio = π_new/π_old` 一旦分母掺入「引擎数值噪声」而非纯粹的「策略漂移」，在 clip range 较窄时会让 clipping 判断失真；
> 3. agentic 多轮轨迹里工具/环境注入的 token 并非模型采样产生，天然没有生成时 logprob 可复用，仍需一次 teacher-forcing 前向补齐。
>
> **`π_ref` 前向则没有这条捷径**：参考模型是冻结的初始/SFT 权重，从第一次策略更新之后就与生成轨迹用的 `π_θ` 不同，其 logprob 不可能是 rollout 生成的副产物，必须专门起一次前向——这是 GRPO KL 项的刚性成本，不像 `π_old` 前向那样可选。

### 边 F — mini-batch 取数（缓冲区 → 训练）

通常经验缓冲区与训练引擎同机/同进程，取 mini-batch 走本地内存，**一般不计入跨网传输**。每迭代分 `n_steps=32` 个 mini-batch 取出，每个 mini-batch（32 prompt×G=16=512 条轨迹）≈ `512×16384×4B ≈ 33.5 MB`；32 个 mini-batch 累计 = 总量 `μ × V_C` ≈ `μ × 1.07 GB`，与边 C 总量一致（μ=1 即整批数据只被取用一遍）。

### 边 G — 训练内部集合通信（FSDP/ZeRO-3）★ 训练侧主导

一次优化器 step 的**带宽最优**每卡通信量 ≈ `3Φ`（前向 all-gather 参数 `Φ` + 反向 all-gather 参数 `Φ` + 梯度 reduce-scatter `Φ`）：

```
每 step 每卡 = 3 × Φ × 2 B = 3 × 141 GB = 423 GB   （BF16；若梯度 FP32，reduce-scatter 段翻倍；与 batch 大小无关，只取决于 Φ）
n_steps = μ × N_p / mini = 1024 / 32 = 32
每迭代每卡 = n_steps × 423 GB = 32 × 423 GB ≈ 13.5 TB / GPU
```

- 该「每卡 ~3Φ」在环形/树形集合通信下**近似与 GPU 数 N 无关**；**聚合上网流量**则 ≈ `N × 3Φ`，随 N 线性增长。
- **关键点**：单 step 通信量只取决于 `Φ`，与该 step 处理多少条轨迹（即 `mini × G`）无关——FSDP 每步都要把完整参数 all-gather 齐、把完整梯度 reduce-scatter 掉，这个量纲只跟模型大小挂钩。所以 `n_steps` 完全由 `N_p/mini` 决定，**与 group size G 无关**（见 §8.2）。
- **额外前向**（不进入优化器 step，仅 all-gather，无反向、无 reduce-scatter），按 `N_p/mini = 32` 个 micro-step 计：
  - 计算 `π_old` logprob：一次全量前向 ≈ `(N_p/mini) × Φ × 2B` = 32 × 141 GB ≈ **4.5 TB/GPU**。
  - GRPO 参考模型前向（KL 项）：再 **4.5 TB/GPU**。
  - 两者合计 **≈ 9.0 TB/GPU**，与优化器 step 本身的 13.5 TB/GPU 属同一数量级，**GRPO 的参考模型开销不可忽略**。

## 6. 关键洞察

1. **跨阶段数据平面完全由权重同步主导。** 真正的「经验数据」（token id + 标量 logprob）加起来约 **1.14 GB**，而权重同步是 **141 GB**——占跨阶段流量的 **99.2%**。权重同步不随 batch 大小变化，而经验数据随 `N_p × G` 线性增长；一旦 `N_p × G` 进一步扩大，两者差距会继续拉大（因为 141 GB 是常数项）。优化 RL 系统的跨阶段带宽 = 优化权重同步。

2. **为什么经验数据这么小？** 因为系统刻意**只传 token id 与「选中 token 的 logprob」**，而不是 hidden states 或 full logits。反事实：若传每 token 的完整 logits——
   ```
   134.2M tokens × V(128256) × 4 B ≈ 68.9 PB
   ```
   比现方案大约 **6400 万倍**。这条设计红线（传 id 不传分布）是 RL 框架能跑通的前提。

3. **训练内部通信才是绝对大头。** 边 G 的 ~13.5 TB/GPU 比整个跨阶段数据平面（142.1 GB）高约 **95×**，这也是为什么并行策略（FSDP vs 3D 并行 TP/PP）、通信-计算重叠、序列打包决定训练吞吐。注意这个量**只取决于 `n_steps = N_p/mini`**，与 group size G 无关——增大 G 不会增加 optimizer 通信次数，只会增加每步处理的 token 数（算力开销）。

4. **GRPO 的参考模型是隐藏成本。** KL 惩罚项要求参考模型对全部 134.2 M 响应 token 做一次全量前向，代价 **4.5 TB/GPU**，加上 536.9 MB 的参考 logprob 传输——这部分开销与 old-logprob 前向同量级，且**不能通过减小 mini batch 摊薄**（`n_steps = N_p/mini` 越大，前向次数越多，通信总量不变，但计算/调度开销上升）。若对训练成本敏感，可考虑降低 KL 系数刷新频率或缓存参考 logprob。

5. **Agentic 的放大点是 `S`（序列长度）而非字节协议。** 多轮工具调用把 `R` 拉长，边 C/E 及边 G 的前向都随 `S` 线性增长（注意力算力随 `S²`）。

## 7. 瓶颈与优化建议

| 维度 | 问题 | 优化 |
|------|------|------|
| 权重同步（A） | 分离部署时 141 GB/步跨集群 | colocated 共卡 + CUDA IPC；FP8 同步；bucket 化 overlap；降低同步频率（受 off-policy 容忍度限制） |
| Rollout 生成 | agentic 长序列 + 长尾轨迹拖慢整批 | 连续批处理、partial rollout、异步 off-policy、长度惩罚 |
| 训练通信（G） | ~13.5 TB/GPU/迭代 | TP+PP 混合并行、序列并行、通信-计算重叠、序列打包去 padding、梯度 BF16 reduce |
| 参考模型（KL） | 额外 4.5 TB/GPU 前向 | 参考 logprob 缓存/低频刷新；降低 KL 系数换取跳过部分前向；固定间隔重算参考模型（★该前向刚性存在，无法像 old logprob 一样直接复用生成期数据） |
| old logprob 前向 | 额外 4.5 TB/GPU 前向 | 复用 rollout 引擎生成时的 logits/logprob，跳过训练侧重算；代价是推理/训练引擎数值路径不同带来的 ratio 噪声，需结合 clip range 权衡（agentic 场景下工具注入 token 仍需补算） |
| 显存 | 优化器状态 ~846 GB | ZeRO-3 分片、CPU offload、优化器状态 8-bit |

## 8. 敏感性分析

固定 `Φ`=70.5B、BF16、μ=1，仅列受影响项。

### 8.1 序列长度 `S`（agentic 多轮的核心变量）

| S (P+R) | 边 C 轨迹 | 边 E old/ref logprob（各一份） | 说明 |
|---------|-----------|-------------------------------|------|
| 16k (P8k+R8k) | 1.07 GB | 536.9 MB | 默认 |
| 32k (P8k+R24k) | 2.15 GB | 1.61 GB | 5 轮左右 agentic |
| 64k (P8k+R56k) | 4.29 GB | 3.76 GB | 长程 agent |

> 权重同步（141 GB）与 `S` 无关，故 `S` 增大时跨阶段流量占比里权重同步的主导性会**逐步下降**；但训练内前向随 `S` 线性、注意力算力随 `S²` 上升。

### 8.2 group size `G`（固定 `N_p=1024`、`mini=32` prompt 不变）

`n_steps = N_p/mini = 32` **不随 `G` 变化**（因为 mini batch 按 prompt 数切分，与每个 prompt 采样几条无关），所以边 G（optimizer 通信）在这张表里是常数；只有边 C/E（经验数据）随 `G` 线性增长：

| G | B（轨迹） | 边 C | 边 E(old, 单份) | n_steps | 边 G/卡（优化器通信，与 G 无关） |
|---|-----------|------|-----------------|---------|-----------------------------------|
| 8 | 8,192 | 537 MB | 268.4 MB | 32 | 13.5 TB |
| 16（默认） | 16,384 | 1.07 GB | 536.9 MB | 32 | 13.5 TB |
| 32 | 32,768 | 2.15 GB | 1.07 GB | 32 | 13.5 TB |

> `G` 越大，组内优势估计越稳定，rollout 生成量、经验回传量（C/E）、单 step 内的 FLOPs 均线性增长；但 optimizer **通信次数与每次通信量不变**（只取决于 `Φ` 和 `N_p/mini`），权重同步 141 GB 也不受影响。

### 8.3 mini batch 大小（固定 `N_p=1024` 不变，单位为 prompt 数）

`mini` 越小，`n_steps` 越多，边 G 总量线性增长（每步仍是固定的 423 GB/GPU）：

| mini（prompt） | 每 mini-batch 轨迹数（×G=16） | n_steps = N_p/mini | 边 G/卡 |
|----------------|-------------------------------|---------------------|---------|
| 16 | 256 | 64 | 27.1 TB |
| 32（默认） | 512 | 32 | 13.5 TB |
| 64 | 1024 | 16 | 6.77 TB |

> mini batch 越大，每步喂入的数据越多、单卡显存压力越大，但换来更少的 optimizer step、更低的通信总量——这是显存与通信量之间的经典权衡。

### 8.4 同步精度

| 精度 | 边 A |
|------|------|
| BF16 | 141 GB |
| FP8 | ~70 GB |

### 8.5 同步频率（off-policy 容忍）

| 每 N 步同步 | 边 A 摊薄 |
|-------------|-----------|
| N=1（on-policy） | 141 GB/迭代 |
| N=4 | 35 GB/迭代 |

## 9. CPU（host）内存峰值

跨阶段/训练内部通信之外，还有一块常被忽略的资源约束：**CPU 内存**。它不跟 GPU 显存共享，一旦打满会直接 OOM-kill 训练进程。峰值高度依赖**是否做了 CPU offload**，分场景给出量级。

### 9.1 不做 CPU offload（纯 GPU 训练，FSDP/ZeRO-3 数据全在显存）

CPU 内存只装“轻量”的东西：

| 用途 | 量级 |
|---|---|
| 经验缓冲区（token id + logprob + reward，即边 C+D+E 之和） | ~1.14 GB |
| Dataloader / prompt 数据集缓存 | 几百 MB ~ 几 GB |
| NCCL pinned memory 通信缓冲区 | 通常几百 MB |
| Python / 框架进程本身开销 | 几 GB |

**总量级：个位数到十几 GB**，不是瓶颈。

### 9.2 优化器状态 CPU offload（ZeRO-Offload / DeepSpeed stage-3 offload）★ CPU 内存暴涨的主因

Adam 优化器状态（FP32 主权重 + 一阶矩 + 二阶矩，梯度视配置可选）搬到 CPU：

```
FP32 主权重: Φ×4B = 282 GB
一阶矩 m:    Φ×4B = 282 GB
二阶矩 v:    Φ×4B = 282 GB
（梯度若也 offload）: Φ×4B = 282 GB
────────────────────────────
合计: 846 GB ~ 1.1 TB
```

ZeRO-3 下这部分是**分片**到所有节点 CPU 上的，单节点 CPU 内存峰值 ≈ `846GB / 节点数`：

| 节点数（每节点 8 卡） | 单节点 CPU 内存峰值（仅优化器状态部分） |
|----------------------|------------------------------------------|
| 2 | ~423 GB |
| 4 | ~212 GB |
| 8 | ~106 GB |
| 16 | ~53 GB |

节点数少（如 1–2 台）时，单机 CPU 内存需求可达**数百 GB 到 1 TB+**，会成为部署的硬约束，需要大内存机型（如 1–2 TB host RAM）。

### 9.3 Rollout 引擎（vLLM/SGLang）的 CPU KV cache offload

agentic 长序列场景常开 CPU offload 存长上下文 KV cache。单条 KV cache：

```
V_kv = 2(K+V) × L × h_kv × S × 2B(BF16)
```

对 Llama-3-70B（GQA，KV head=8 → h_kv=1024）、`S=16k`：

```
V_kv = 2 × 80 × 1024 × 16,384 × 2B ≈ 5.37 GB / 条
```

并发几十到上百条长序列同时 offload 到 CPU 时，量级可达**几百 GB**（如 64 条并发 ≈ 344 GB）。

### 9.4 Agentic 工具调用引入的 CPU 内存 ★ 与 offload 无关的独立来源

前三节的内存需求都能从 `Φ`、`B`、`S` 等模型/batch 参数**解析算出**；但 agentic 场景特有的**工具调用**，其内存需求由**工具执行的并发进程数与单进程内存**决定，与模型大小无关，更像是运维层面需要压测确定的变量，而不是解析量：

| 工具类型 | 单实例内存 | 主要驱动因素 |
|---------|-----------|-------------|
| 代码执行沙箱（Python 解释器/容器，可能加载 numpy/pandas 等库） | ~100–500 MB（空载），加载数据/建模后可到 GB 级 | rollout 引擎的**并发上限**（远小于总轨迹数 B），不是 B 本身 |
| 本地检索索引（FAISS/BM25，供搜索工具用） | 几十 GB（小语料）~ 几百 GB（网页级语料） | 语料规模，**常驻**、不随并发变化 |
| headless 浏览器（网页浏览工具） | ~100–500 MB / 实例 | 并发实例数 |
| 工具返回结果缓存 | 通常与边 B'/C 同量级（文本） | 若返回结构化大数据（整表、图片）会显著更大 |

**估算示例**：假设 rollout 引擎设置 512 路并发，其中 30% 触发代码沙箱（平均 300 MB/个），另加载一个 50 GB 的检索索引：

```
代码沙箱: 512 × 0.3 × 300 MB ≈ 46 GB
检索索引（常驻）: 50 GB
────────────────────────
合计 ≈ 96 GB
```

**关键区别**：优化器/KV cache 的 offload 内存需求由 `Φ`、`S` 等**确定性参数**推导，可以在设计阶段精确计算；工具调用的内存需求由**并发度、工具实现细节**决定，同样的模型和 batch 配置下，换一个代码沙箱实现（进程 vs 容器 vs 轻量 WASM 沙箱）或换一个检索后端，内存量级可以相差一个数量级——需要**实测压测**而非纯解析估算，只能给出数量级参考。

### 9.5 汇总

| 场景 | CPU 内存峰值量级 | 是否可解析计算 |
|------|-------------------|----------------|
| 无 offload | 十几 GB（可忽略） | 可精确算 |
| 优化器状态 offload（少节点，如 1–2 台） | 数百 GB ~ 1 TB+ | 可精确算（取决于 Φ、节点数） |
| 优化器状态 offload（多节点分片，如 8–16 台） | 数十 ~ 百余 GB / 节点 | 可精确算 |
| Rollout KV cache offload（高并发长序列） | 数百 GB | 可精确算（取决于 S、并发数） |
| Agentic 工具调用（沙箱/检索/浏览器） | 数十 ~ 数百 GB | **需压测**，仅数量级参考 |

> **结论**：不做任何 offload，CPU 内存峰值是十几 GB 量级，可忽略；一旦做优化器状态或 KV cache 的 CPU offload，峰值会跳到**百 GB 到 1 TB+ 量级**，且强烈依赖节点数/并发数；agentic 工具调用则引入了**第三类、且无法纯解析预测**的 CPU 内存开销，取决于工具实现和并发配置，实际部署前应做压测而非只靠公式估算。

## 10. 一句话总结

> 在 Llama-3-70B 的 agentic GRPO 后训练里（1024 个 prompt × group size 16 = 16,384 条轨迹/迭代，mini batch=32 个 prompt → 32 个 optimizer step），**跨阶段传输 ≈ 142.1 GB/迭代，其中 99.2% 是权重同步**；真正的经验数据（token id + 标量 logprob）仅约 1.14 GB，因为系统只传 token id 而非分布（否则会是 68.9 PB）。**训练引擎内部 32 个 optimizer step 累计的集合通信 ~13.5 TB/GPU/迭代**，比跨阶段流量高约 95 倍，是吞吐的真正瓶颈；GRPO 的参考模型 KL 项额外贡献 4.5 TB/GPU 的隐藏前向开销，与优化器 step 本身同量级，是不可忽视的成本项。
