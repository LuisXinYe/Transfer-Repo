# Agentic RL 后训练可行性分析：内存峰值与带宽瓶颈判定

> 模型/场景/并行/硬件配置**复用** [`agentic_rl_npu_analysis.md`](./agentic_rl_npu_analysis.md)：**Qwen3-235B-A22B**（MoE 235B/22B激活，94层，h=4096，128专家top-8，GQA 64Q/4KV，FP8）· GRPO 共卡 on-policy · `N_p=1024, G=16, B=16384, mini=32, n_steps=32` · 三场景 S=16k/32k/128k（文本/图片/视频）· 并行 `N=128, EP=32(4专家/卡), 训练ZeRO-3 / 推理DP32×TP4` · 硬件 A2(910B) / A3(910C) / 950DT。
> 本文回答两个问题：**① 内存（HBM / CPU DRAM）按峰值口径够不够**（DRAM 部分细致建模沙箱）；**② 在给定带宽配置下，各阶段通信是否构成瓶颈**——判据是**相对值**（能否被计算掩盖 τ、相对端到端占比 ρ），而非绝对传输时间。
>
> **⚠️ 单位口径（全文按 die 计）**：并行分解 `N = TP×CP×PP×DP = 128` 的 **128 = die（=rank）数**，非封装数。各代封装内 die 数不同 → **物理卡（封装）数：A2 = 128 卡、A3 = 64 卡、950DT = 64 卡**。**per-rank（per-die）规格三代对齐**：A2 die = A3 die 完全相同（**HBM 64GB、HBM BW 1.6TB/s、scale-up 392GB/s bidi = 196 uni**，A3 封装两 die = 784 bidi）；950DT die = **HBM 72GB、HBM BW 2.0TB/s、scale-up ~2TB/s bidi**。故 **HBM 判定中 A2 与 A3 两列相同**；A3 相对 A2 的唯一优势是 **scale-up fabric 域更大**（超节点 UB 可容纳 EP=32 的 32-die 域，A2 的 8-die 岛容不下 → 借道 RoCE）。**τ 是比值，对 die/封装口径不变**（通信 BW 与计算聚合同步 ×2）；仅 **HBM 容量与 HBM 带宽**按 die 收紧。

## 0. 方法总览：峰值内存 + 相对带宽判据

| 维度 | 口径 | 判定方式 |
|------|------|----------|
| **HBM** | **单 die 峰值**：训练相 / 推理相分别核算，共卡分时取 `max(两相)` | 峰值 vs **per-die 容量(A2 64 / A3 64 / 950DT 72 GB)** |
| **CPU DRAM** | **单节点峰值**：同一时刻共存的所有来源相加（缓冲区 + offload + **沙箱** + OS） | 峰值 vs ~1.5TB/节点 |
| **带宽** | **相对判据**，不看绝对传输时间 | **τ = T_comm/T_compute**（能否被计算掩盖）；**ρ = T_未掩盖/T_e2e**（相对端到端是否可忽略，T_e2e 含序列化/Ray 开销） |

带宽判定的两条原则（用户思路的形式化）：
1. **通信时间长 ≠ 瓶颈**——若同阶段的计算更长且可重叠，通信被掩盖，加带宽无收益（判据 τ）。
2. **即便某段通信无法掩盖，若它相对端到端时间可忽略，也无必要提升带宽**（判据 ρ）——其中端到端时间必须包含样本数据在 CPU↔GPU 搬运时的**序列化/反序列化与 Ray 静态开销**，因为对经验数据面而言，这些软件开销远大于线上传输本身。

# Part 1 · 内存可行性（峰值）

## 1. HBM 诉求（NPU 单卡峰值）

### 1.1 口径

共卡分时复用：rollout 时释放梯度/优化器给 KV cache，training 时释放 KV 给激活。**单卡 HBM 峰值 = max(训练相, 推理相)**，两相分别核算。

### 1.2 训练相峰值 = 静态 + 激活

静态（与场景无关）：专家分片 14.2GB(BF16) + 非专家 ZeRO 常驻 0.1 + 梯度 3.7 + 优化器 22.0 ≈ **40 GB/卡**。
激活（1 序列/micro-batch + 激活重算，存层边界残差流）：`n_layers × S × h × 2B`。

| 场景 | 静态 | 激活 | **训练相峰值** | A2(64) | A3(64) | 950DT(72) |
|------|------|------|---------------|--------|--------|-----------|
| 文本 16k | 40 | 12.6 | **~53 GB** | ✅(紧) | ✅(紧) | ✅ |
| 图片 32k | 40 | 25.2 | **~65 GB** | ❌(略超) | ❌(略超) | ✅(紧) |
| 视频 128k | 40 | 101 | **~141 GB** | ❌ | ❌ | ❌(远超72) |

> 注：本表为**基线 P0（TP4、无 SP）**，激活未被 SP 切；图片/视频的超限在 §1.5 由 memory-first 并行（P1 开 SP → 图片训练降到 ~46GB；P2 → 视频降到 ~53GB）修复。**A2 与 A3 因 per-die 容量相同（64GB），判定完全一致。**

### 1.3 推理相峰值 = 权重底座 + KV cache

权重底座（FP8，EP 分片专家 7.1 + TP 分片非专家 2.0）≈ **9 GB/卡**。
KV（GQA h_kv=512，TP=4 切分）：0.79 / 1.58 / 6.30 GB/序列/卡；并发 b=32/组：

| 场景 | 底座 | KV(b=32) | **推理相峰值** | A2(64) | A3(64) | 950DT(72) |
|------|------|----------|---------------|--------|--------|-----------|
| 文本 | 9 | 25.3 | **~34 GB** | ✅ | ✅ | ✅ |
| 图片 | 9 | 50.6 | **~60 GB** | ✅(紧) | ✅(紧) | ✅ |
| 视频 | 9 | 201.6 | **~211 GB** | ❌ | ❌ | ❌（b=8→~59GB，三代均 ✅紧；见 §1.5 P2） |

### 1.4 HBM 判定小结

- **文本**：峰值由训练相主导（53 vs 34），三代可行（A2/A3 均紧、950DT 松）。
- **图片**：基线训练相 65GB **A2、A3 均略超 64**（两者 per-die 相同）、950DT(72) 紧过；开 SP（P1）后降到 ~46GB → 三代 ✅。
- **视频**：**两相都破**——训练激活 101GB 需序列并行切激活，推理 KV 需降并发(b→8/16)；基线三代（含 950DT，211>72）都放不下，必须 memory-first（§1.5 P2：训练 53 / 推理 59，三代 ✅紧）。
- **结论**：HBM 峰值的敏感变量是 **S**（激活 ∝S、KV ∝b×S）；per-die 64/64/72 下 **A2 与 A3 判定一致**，950DT 仅多 8GB 余量，**代际差异不在 HBM 容量而在 scale-up（§4）**。

## 1.5 并行配置比较（memory-first：先放下，再看带宽趋势）

**动机**：§1.4 显示**视频 128k 两相都破 HBM**（训练激活 101GB、推理 KV 211GB@b32）。基线并行 `EP32×TP4` 只能靠"降 b + KV offload"勉强塞下，代价是把压力转嫁到 CPU DRAM（§2.5 最坏堆叠逼近 1.4TB）。本节换思路：**先用并行度把 HBM 放下（memory-first），再看这么做对带宽诉求的变化趋势**。

### 1.5.1 并行工具箱：各维度切什么、加什么通信

| 维度 | 切分对象 | 训练相内存效果 | 推理相内存效果 | 新增 fabric 通信 |
|------|----------|---------------|---------------|-----------------|
| **EP=32**（基线） | 专家 MLP | 专家权重 /EP | 专家底座 /EP | all-to-all 12.3MB/token（基线开销） |
| **TP=t** | h 维（注意力+dense） | 注意力/MLP 内部激活 /t | **KV /t**、底座 /t | all-reduce ∝ tokens·h/层（**尽量压在节点内**） |
| **SP**（Megatron 序列并行；**度数 = TP，叠在 TP 组内、0 额外占卡**） | **层边界残差流/LN/dropout 激活** | **残差激活 /SP = /TP**（TP 单独切不动的那部分） | 用 RS+AG **替换** TP 的 all-reduce，**通信量≈不变** |
| **CP=c**（上下文并行，Ring/Ulysses，**额外占卡**） | 序列 S | 激活 /c、prefill KV /c | decode KV /c | **序列 all-to-all/ring ∝ S·h/层（跨节点则抬高诉求）** |
| **PP=p** | 层 | 每 stage (静态+激活)/p | 每 stage 底座/KV /p | 仅**边界 P2P**（S·h，廉价）+ **流水 bubble ↓MFU** |
| **ZeRO-3** | 优化器/梯度/参数态 | 优化器+梯度 /N | — | 逐层 param all-gather |

**关键机制（回答"为什么要开 SP"）**：标准 Megatron **TP 单独切时，层边界的残差流/LayerNorm/dropout 激活是跨 TP rank 复制的**——所以 §1.2 的 101GB 激活在 TP4 下并**没有** /4。**Megatron 序列并行（SP）专门把这部分残差激活也按 TP 切**，且用 reduce-scatter + all-gather **替换** TP 原本的 all-reduce，**通信总量几乎不变**。故"开 SP"是**近乎免费的降内存开关**。

### 1.5.2 五套配置对视频 128k 的 HBM 判定（N=128，EP=32）

**卡预算约束**：128 **die** 按 `N = TP × CP × PP × DP = 128` 划分注意力/dense 侧（物理卡：A2 128 / A3 64 / 950DT 64 封装）;**EP=32 是专家轴**(专家 MLP 的 all-to-all 组)。**SP 不占卡**——SP 度数恒 = TP 度数,只是把 TP 组内本来复制的残差激活也切掉。下表每套配置给出**完整度数元组**;训练相激活 `∝ 101/(SP×CP) = 101/(TP×CP)`,推理相 KV `∝ 6.3·b/(TP×CP)`。

**判定容量：per-die A2 64 / A3 64 / 950DT 72 GB（A2=A3）**。

| 配置 | **度数元组**（EP=32 固定） | 视频**训练相** | 视频**推理相** | 需 KV offload? |
|------|--------------------------|---------------|---------------|----------------|
| **P0 基线** | `TP4·SP✗·CP1·PP1·DP32` | 40 + 101 = **141 ❌**（三代） | 9 + 6.3·b：b8→59 / **b32→211 ❌** | **是**（且须降 b） |
| **P1 开 SP** | `TP4·`**`SP4`**`·CP1·PP1·DP32` | 40 + 101/4 = **~65**：A2/A3 ❌、**950DT ✅紧** | 同 P0（SP 不改 KV，b32→211 ❌） | 是（KV 仍需 b8） |
| **P2 TP↑+SP** ★ | **`TP8·SP8`**`·CP1·PP1·DP16` | 40 + 101/8 = **~53 ✅**（三代）¹ | 9 + 3.15·b：**b16→59 ✅紧**（三代）/ **b32→110 ❌** | **否**（须 b≤16） |
| **P3 +CP** | `TP4·SP4·`**`CP4`**`·PP1·DP8` | 40 + 101/16 = **~46 ✅**（三代） | KV/(TP·CP)：b32→~30 ✅（三代） | 否 |
| **P4 +PP** | `TP2·SP2·CP1·`**`PP4`**`·DP16` | 每 stage 静态~10 + 激活（micro-batch 可调）**✅²** | 底座/KV /PP | 否（仅训练相） |

> 度数读法：`TP4·SP4` 表示 TP 组内(4 die)开序列并行(SP=TP=4);`CP4` 是**额外**占 4 路 die 的上下文并行;`PP4` 占 4 路流水级。各元组均满足 `TP×CP×PP×DP=128`(die)。

¹ **EP 与 TP 的卡账（per-die 64GB 下更关键）**：EP=32 的专家 all-to-all 组与 TP 组正交,`EP×TP_expert ≤ 128`。TP8 下若专家不再按 TP 切,则 EP 需降到 16(专家底座 +~14GB → 训练 **~67GB**)——**67>64,A2/A3 均 ❌**(仅 950DT 72 勉强)。故 per-die 口径下**必须让框架(MindSpeed/Megatron-MoE)支持专家侧 ETP 以维持 EP=32(专家 /32 → 训练 53 ✅)**;ETP 从"可选优化"变成 **A2/A3 上的必需项**。**须按目标框架的 EP-TP 组合能力校验**。
² PP 引入流水 bubble 压低 MFU,且**与多轮 RL rollout / 共卡 decode 路径不兼容**(轨迹变长、rollout 不像训练那样能流水),故 PP 仅作**训练相**辅助;其激活省不省取决于 1F1B 在飞微批数,需调 micro-batch。

### 1.5.3 对带宽诉求的影响趋势（本节核心）

**memory-first 的落地顺序与带宽代价**：

1. **先开 SP（P1，SP=TP=4）——免费降内存**：把 TP 组内本来复制的残差激活也切了(/4)，视频训练 141→65GB 放下，**通信量≈不变**（RS+AG 替换 all-reduce）→ **带宽诉求持平**。第一优先、零带宽代价、0 额外占卡。
2. **再抬 TP 到 8（P2，TP8·SP8·DP16）——同时解决推理 KV，且免 offload**：激活 /8、KV 也 /2 → 视频推理 b16 仅 59GB（<64，三代 **✅紧**），**不再需要 KV offload**。副作用是把更多 all-reduce/RS/AG 流量压到**同一 scale-up 域内**——TP8 = 8 个 die（A3 = 4 封装 / A2 = 8 封装 = 一个节点岛），**被 UB 吸收**（per-die 196GB/s uni）；**跨节点的 EP all-to-all 通信量不变 → §4 的 EP τ 判定不变**。**A3 仍全程 ✅**（惟余量按 64GB 计变紧，需 ETP 保 EP=32，见 ¹）。
   - 附带收益：P2 消除视频 KV offload → **§2.5 的 DRAM 最坏堆叠（~1.4TB）随之显著回落**（去掉 400–800GB 的 offload 项）——memory-first 在 **HBM 和 DRAM 两侧同时受益**。
3. **CP 是唯一真正抬高跨节点带宽诉求的杠杆（P3）**：只有当 S 超出 TP-SP 能切的范围（或 TP 已顶满节点 8 卡）才需要 CP；它新增的**序列 all-to-all/ring ∝ S·h 会吃掉 A3 的 4–6× 富余**。趋势：**用 CP 来 memory-first ⇒ 带宽诉求上升**。
4. **PP 用 MFU 换内存（P4）**：通信廉价（仅边界 P2P），但 bubble 压低 MFU（等价抬高 τ），且不适配共卡多轮 rollout——**训练相专用**。

**A2 的结论不因并行调整而改变**：无论怎么切，A2 的 EP=32 仍跨 4 个 8 卡岛、~77% 字节借道 RoCE 25GB/s（§4.1），**并行配置救不了 A2 的 comm-bound**——只有**超节点 fabric**能救。这反过来印证 §4 的结构性判断。

**一句话趋势**：**放得下 ≠ 更费带宽**。memory-first 的正确路径是 **开 SP（免费）→ 抬 TP 到节点内 8（把流量塞进 UB、免 offload）**——此路径下 **A3 的带宽诉求基本持平、EP τ 不变**；只有被迫上 **CP（跨节点序列并行）** 时带宽诉求才真正上升，而 **PP 是拿 MFU 换内存的训练相开关**。

¹ per-die 64GB 下视频 59GB 仅 ✅紧,且**取决于 EP**：EP32（需 ETP）→53/59 ✅ / EP16→+14GB→67 ❌（A2/A3 同）。故 **ETP 是 A2/A3 的必需项**。另：A2 的 scale-up 本就 ❌（comm-bound）→ **A2 综合仍不可行**，此 HBM ✅紧 仅指显存账。

## 2. CPU DRAM 诉求（单节点峰值，含沙箱细致建模）

### 2.1 口径

单节点 = 8 卡（封装）+ ~1.5TB DRAM。**本节按 A2 口径 = 128 封装 / 8 = 16 节点**，`C_traj = 1024/16 = 64`。

> **⚠️ per-die 口径下的节点数级联（待定）**：128 die 时 **A3 = 64 封装**，若每节点仍 8 封装则**只有 8 节点** → 每节点并发轨迹 `C_traj = 1024/8 = 128`（**翻倍**），沙箱 DRAM 峰值随之约 ×2；但 A3/950DT 服务器每节点的封装数与 DRAM 容量取决于具体 SKU（CloudMatrix 级密度可能更高）。本节沙箱/堆叠数值仍按 16 节点（A2）给出;**A3/950DT 需按目标服务器的"封装数/节点、DRAM/节点"重代 `C_traj` 与峰值**——节点越少、每节点越密,沙箱峰值越高。

### 2.2 来源 A：经验缓冲区（方案①：token id + logprob + 压缩帧）

| 场景 | 全集群 | 每节点(÷16，分片存) |
|------|--------|---------------------|
| 文本 | ~2.5 GB（id+logprob+reward） | ~0.2 GB |
| 图片 | ~55 GB（+16384×~3.2MB 帧） | ~3.4 GB |
| 视频 | ~170 GB（+16384×~10MB 片段） | ~11 GB |

> 若缓冲区**不分片**（基线 driver 汇聚），全量压在 driver 单节点——视频 ~170GB 单点，是 §5 数据面问题的存储侧镜像。

### 2.3 来源 B：offload

| 项 | 触发条件 | 每节点 |
|----|----------|--------|
| 优化器状态 offload | **可选**（22GB/die；能否留 HBM 取决于配置：P2 训练 53<64 可留 → 不 offload；紧配置才 offload。**per-die 下 A2=A3 同判，非芯片相关**） | 2.82TB/16 ≈ **176 GB** |
| 视频 KV offload | 视频必选（HBM 放不下） | b=8：50GB/卡×8 ≈ **400 GB**；b=16：~800 GB |

### 2.4 来源 C：沙箱（细致建模）★

**为什么要细致建模**：沙箱内存不由模型参数决定，由**工具执行的并发进程数 × 单进程内存**决定；且 agentic RL 的 rollout 是**波同步**的（1024 条轨迹同波起跑），沙箱的内存尖峰会**相关性叠加**——必须按峰值记账，不能按平均。

**变量定义**：

| 变量 | 含义 | 取值来源 |
|------|------|----------|
| `C_traj` | **每节点并发轨迹数** = 每时刻并发生成的轨迹数 ÷ 节点数 = `(DP_infer×b)/n` | **文本 64 / 视频 32~16**（见下推导） |
| `f_sb` | 触发沙箱的轨迹比例 | 任务型：coding≈1，混合≈0.3–0.7 |
| `M_base` | 单沙箱稳态 RSS | 空载容器 0.1–0.3GB；载入 numpy/pandas 后 0.5–1.5GB |
| `M_spike` | 单沙箱峰值 RSS（跑测试/编译/并行 pytest） | 1–4GB |
| `κ` | **波同步相关系数**（多少沙箱同时处于尖峰） | 0.3–1；RL 波同步下取 0.5–0.8 |
| `M_shared` | 节点常驻共享项（检索索引、模型缓存） | 0–200GB |

**`C_traj` 的正确推导（澄清 `1024/16` 的来历）**：
- 总轨迹数 `B = N_p × G = 1024 × 16 = 16384`（**每 prompt 采样 G=16 条轨迹**）。
- **并非同时在跑**：同一时刻并发生成的轨迹数受推理批容量约束 = `DP_infer × b`（文本 32×32 = 1024），故 B 分 `16384 / 1024 = 16` 波（wave）跑完。
- 每节点并发沙箱 `C_traj = (DP_infer × b) / n = 1024 / 16 = 64`。
- **口径澄清**：分子 1024 = `DP×b`（推理并发容量，**恰好等于 N_p=1024，纯属巧合**）；分母 16 = **n（节点数）**，**不是 G**。**G=16 决定的是总量 B 与波数（16 波），不直接进入"每时刻并发"**。原式 `1024/16` 数值正确，但两个操作数都极易被误读为 `N_p/G`——特此更正。
- **关键耦合**：`C_traj` 由 **b** 决定，而 b **同时**是推理 KV 的 HBM 旋钮——**沙箱 DRAM 与 KV-HBM 共用同一个旋钮 b**。视频为放下 KV 把 b 压到 16/8，则并发沙箱数同步降到 32/16：

| 场景 | b | 并发容量 `DP×b` | 波数 `B/(DP×b)` | **每节点 `C_traj`** |
|------|---|----------------|-----------------|---------------------|
| 文本 | 32 | 1024 | 16 | **64** |
| 视频（b=16） | 16 | 512 | 32 | **32** |
| 视频（b=8） | 8 | 256 | 64 | **16** |

> 下表沙箱峰值按**文本 b=32 的 64 并发上界**保守计；视频因 b 更小、并发更低，沙箱 DRAM 实际更松（§2.5 视频行沿用 64 口径，属高估的保守侧）。

**峰值公式**：

```
DRAM_sandbox(节点) = C_traj × f_sb × [ M_base + κ·(M_spike − M_base) ] + M_shared + 页缓存
```

**代入四类 agent（每节点 64 并发；文本 b=32 基准/上界）**：

| Agent 类型 | f_sb | M_base | M_spike | κ | **每节点沙箱峰值** |
|-----------|------|--------|---------|---|--------------------|
| 轻工具（计算器/搜索文本） | 0.3 | 0.2 | 0.2 | — | **~4 GB**（可忽略） |
| **Coding（中等，解释器+科学库）** | 1.0 | 0.8 | 2.5 | 0.5 | **~106 GB** |
| **SWE 重型（仓库+依赖+pytest）** | 1.0 | 1.5 | 4.0 | 0.8 | **~224 GB**（+仓库页缓存 ~50GB → **~280 GB**） |
| GUI（headless 浏览器） | 1.0 | 0.5 | 0.8 | 0.5 | **~42 GB** |
| 检索索引（若搜索工具带本地索引） | — | — | — | — | **+50–200 GB 常驻**（M_shared） |

**三个建模要点**：
1. **波同步放大 κ**：同波轨迹几乎同时进入"跑测试"阶段 → 尖峰对齐。异步/partial rollout 可打散（κ→0.3），是软件侧降峰手段。
2. **沙箱寿命模型**：per-trajectory 持久会话（SWE 式，repo checkout 常驻）取 `f_sb≈1`；per-call 瞬态沙箱按占空比折减。上表按持久会话（保守）。
3. **不可解析性**：`M_base/M_spike` 强依赖工具实现（进程 vs 容器 vs WASM、依赖集合），上表是量级参考，**上线前必须压测**。

### 2.5 单节点 DRAM 峰值叠加与判定

叠加口径：缓冲区 + offload + 沙箱 + OS/框架(~50GB)。取代表组合：

| 场景组合 | 缓冲区 | KV offload | 优化器 offload | 沙箱 | OS | **合计** | vs 1.5TB |
|----------|--------|-----------|----------------|------|----|---------|----------|
| 文本 + coding | 0.2 | — | — | 106 | 50 | **~156 GB** | ✅ 10% |
| 图片 + GUI | 3.4 | — | — | 42 | 50 | **~95 GB** | ✅ 6% |
| 视频 + 轻工具 | 11 | 400(b=8) | — | 4 | 50 | **~465 GB** | ✅ 31% |
| 视频 + SWE 重 | 11 | 400 | — | 280 | 50 | **~741 GB** | ✅ 49% |
| 视频 + SWE 重 + 优化器 offload（紧配置） | 11 | 400 | 176 | 280 | 50 | **~917 GB** | ⚠️ 61% |
| **最坏堆叠**（b=16 offload + SWE + 索引 100G + 优化器 offload） | 11 | 800 | 176 | 380 | 50 | **~1.42 TB** | ❌ 逼近 95% |

> **per-die 修正**：A2 与 A3 的 die 同为 64GB，**DRAM 画像不再随芯片区分**（原表"A2 叠优化器 offload / A3 不叠"的差异已取消）。优化器 offload 是否发生由**并行配置**决定——P2 训练 53<64 可把 22GB 优化器留 HBM → 不产生此项；只有更紧的配置才 offload，**A2/A3 同此判断**。

**DRAM 判定小结**：
- 常规组合 **1.5TB ✅ 够用**（占用 10%–50%）；
- **沙箱是 CPU DRAM 的头号不确定变量**：轻工具可忽略（GB 级）→ SWE 重型 ~280GB/节点（与 KV offload 同量级）；
- **危险组合是"视频高并发 KV offload × 重型沙箱 × 优化器 offload"同时堆叠**——逼近 1.5TB（优化器 offload 由紧配置触发，A2/A3 同判，非芯片相关）。缓解开关：降 rollout 并发（KV offload 线性降）、异步 rollout 打散 κ、沙箱换轻量实现（WASM/进程池）、索引放独立节点。
- **首选缓解是 §1.5 的 memory-first 并行（P2：TP8+SP）**：它把视频 KV 塞回 HBM、**彻底消除 400–800GB 的 KV offload 项**，本表视频行的 offload 列直接归零，最坏堆叠从 ~1.4TB 回落到 ~0.6TB 量级——**HBM/DRAM 两侧同时受益，且带宽诉求不升**（见 §1.5.3）。

# Part 2 · 带宽可行性（相对判据）

## 3. 判据框架：τ（计算掩盖）与 ρ（端到端占比）

### 3.1 定义

对每个阶段（prefill / decode / logprob 前向 / 训练 fwd+bwd / 数据面）：

```
τ = T_comm(该阶段通信) / T_compute(与之可重叠的计算)
ρ = T_exposed(未被掩盖的通信) / T_e2e(整迭代端到端)

T_e2e = T_rollout + T_logprob + T_train + T_dataplane(+ T_reward)
T_dataplane = D2H/H2D 线上传输 + 序列化/反序列化 + Ray RPC/object store 静态开销
```

**判定规则**：

| τ | 结论 | | ρ | 结论 |
|---|------|---|---|------|
| < 0.5 | ✅ 通信被掩盖，**加带宽无收益** | | < 5% | ✅ 相对端到端可忽略，**无必要提升带宽** |
| 0.5–1 | ⚠️ 勉强掩盖，依赖重叠工程（DualPipe 类） | | 5–15% | ⚠️ 值得优化——但优化对象通常是**软件路径**而非带宽 |
| > 1 | ❌ comm-bound，加带宽/换 fabric 才有效 | | > 15% | ❌ 显著拖慢端到端 |

**两个 regime**：带宽型阶段（prefill/logprob/训练——大消息批量）用 τ 判；**decode 是延迟型**（每 token 串行 94 层×2 次小消息 all-to-all），看每步的通信耗时能否藏进该步的计算/KV 读之下。

### 3.2 一个关键性质：τ 与场景近似无关

all-to-all 通信与 matmul 计算都 ∝ tokens=B×S，**分子分母同步涨 → τ 稳定**；视频的注意力 ∝S² 让计算涨更快 → τ 反而更小（通信更好藏）。所以「要不要加卡间带宽」是**代际固有属性**，按文本场景判定即可外推。

### 3.3 端到端时间参考（A3，用于算 ρ）

计算侧（40% MFU，仅激活 22B 参与）：

| 场景 | prefill | decode | logprob×2 | 训练 | 计算侧合计 |
|------|---------|--------|-----------|------|-----------|
| 文本 | 144 s | ~1035 s（KV 读限，见 §6） | 577 s | 865 s | **~2620 s** |
| 图片 | ×2 ≈ 3170 s（三项）+ decode ~2070 s | | | | **~5240 s** |
| 视频 | ×8 + 注意力 S² 加成 ~30–50% | decode 2000–6000 s（KV offload 后受 PCIe 拖累） | | | **~1.9–2.2 万 s** |

## 4. scale-up 带宽：EP all-to-all 逐阶段判定

### 4.1 每阶段通信量与 τ（文本；单卡·每迭代）

all-to-all 单元：`2·top_k·h·2B = 128KB/token/层` × 94 层 = **12.3 MB/token（一次前向）**。

| 阶段 | 单卡通信 | T_comm→ | **A2**(有效32.5GB/s¹) | **A3**(392GB/s) | **950DT**(2TB/s) |
|------|----------|---------|----------------------|------------------|-------------------|
| Rollout prefill | 12.9 TB | | 398s / 计算308s → **τ=1.29 ❌** | 33s/144s → **τ=0.23 ✅** | 6s/115s → **τ=0.06 ✅✅** |
| old+ref logprob 前向 | 51.7 TB | | 1590s/1230s → **τ=1.29 ❌** | 132s/577s → **τ=0.23 ✅** | 26s/461s → **τ=0.06 ✅✅** |
| 训练 fwd+bwd | 51.7 TB | | 1590s/1845s → **τ=0.86 ⚠️**² | 132s/865s → **τ=0.15 ✅** | 26s/692s → **τ=0.04 ✅✅** |

¹ A2 的 EP=32 跨 4 个 8 卡岛，~77% 字节走 RoCE 25GB/s → 有效 ≈32.5GB/s。
² A2 训练 τ=0.86 是"带宽账"上的勉强及格；叠加 RoCE 高延迟、188 次/步同步、专家负载不均长尾后实际 >1——纯前向阶段（τ=1.29）已明确越线。

> **per-die 口径与 τ 不变性**：全文 128 = die。**per-die scale-up：A2 die = A3 die = 392 GB/s 双向（= 196 单向）**（每 die 集成 7×224Gbps UB SerDes）；A3 双 die 封装的 784 双向是**封装级**（=2×392 die），单个 rank（die）只能用自己那份 392 bidi。**本表 A3 列的 392（单向）与计算侧时间是"封装等效"口径**；换成 per-die，则通信 BW 减半（196）、计算聚合也减半（时间 ×2）——**τ = 通信/计算 是比值，分子分母同步 ×2 → τ 值与判定完全不变**（prefill 0.23、训练 0.15…照旧）。950DT die ≈ 1 TB/s 单向（封装 2TB/s ÷ 2 die）。
> **A2 vs A3 的真正差异 = fabric 域大小,非 per-die 带宽**：两代 per-die 带宽相同（196），但 **A2 的 UB 岛只有 8 die**，EP=32 的 32-die 域跨 4 岛、~77% 字节借道 RoCE 25GB/s → 有效仅 **32.5GB/s → τ>1 ❌**；**A3 的超节点 UB 域可容纳 32-die**，EP=32 全程 on-UB（196/die）→ τ=0.23 ✅。**代际差在 scale-up fabric 能否装下 EP 域,而非单 die 快慢。**

### 4.2 decode（延迟型）：每步预算

每 decode 步（全局 1024 并发）：每卡通信 ~99MB，拆成 **188 次同步 all-to-all**（94层×dispatch/combine）：

（per-die 口径：通信 BW 部分按 196GB/s，KV 读按 1.6TB/s）

| | 通信/步（BW+延迟） | 该步可掩盖预算（max(计算, KV读)） | 判定 |
|---|---------------------|-----------------------------------|------|
| A2 | 3.0ms + 188×25μs ≈ **7.7 ms** | max(2.4, 15.8) = 15.8 ms | ⚠️ 名义可藏，但同步长尾极敏感 |
| A3 | 0.5ms + 188×3μs ≈ **~1.1 ms** | max(2.4, 15.8) = 15.8 ms | ✅ τ≈0.07 |
| 950DT | 0.1ms + 188×2μs ≈ **~0.5 ms** | max(1.8, 12.65) = 12.65 ms | ✅✅ |

### 4.3 卡间带宽结论

- **A2 ❌**：纯前向阶段 τ≈1.3（comm-bound），训练 τ≈0.86 且延迟/长尾雪上加霜——**瓶颈成立**。但解法不是渐进加带宽，而是**把 EP 域收进超节点级 fabric**（结构问题）。
- **A3 ✅**：全部阶段 τ=0.15–0.23，**富余 4–6 倍**；decode 通信藏在 KV 读之下。**互联带宽不是瓶颈，加带宽无收益**（余量足以覆盖 MFU 提到 60%、或重叠效率打七折的情形）。
- **950DT ✅✅**：τ=0.04–0.06，富余 ~20 倍。
- **场景外推**：τ 与 S 近似无关（§3.2），视频更宽松——三场景同判。

## 5. 网络/主机带宽（scale-out + host）：经验数据面与序列化/Ray 开销

### 5.1 原始字节量：任何代际都是零头

经验数据（边 C/E/F，含压缩帧）全集群每迭代：文本 ~2.5GB / 图片 ~55GB / 视频 ~170GB。摊到 16 节点、RoCE 25–50GB/s：**线上传输秒级**（视频 170GB÷16节点÷25GB/s ≈ 0.4s/节点）。**看原始带宽，网络永远不是瓶颈。**

### 5.2 真实开销在软件路径：序列化/反序列化 + Ray + driver 汇聚

参数化模型：

```
T_dataplane ≈ n_transits × V_batch / BW_serdes + N_rpc × t_rpc
  n_transits：样本张量经 driver 收发的次数（基线 ≈10 次；TQ 后 ≈0，见 TQ 分析报告）
  BW_serdes：序列化有效吞吐（driver 单进程 pickle+concat ≈1–2 GB/s；分布式 worker 并行可 ×8–16）
  N_rpc×t_rpc：Ray 调用静态开销（每阶段 fan-out W_agent/R_*，全迭代 ~10³ 次 × 1–5ms ≈ 秒级）
```

代入（基线无 TQ，n_transits=10，BW_serdes=1.5GB/s）：

| 场景 | V_batch | **T_dataplane（基线）** | T_e2e（§3.3） | **ρ** | 判定 |
|------|---------|--------------------------|---------------|-------|------|
| 文本 | 2.5 GB | ~17 s + Ray ~秒 ≈ **~20 s** | ~2620 s | **0.8%** | ✅ 可忽略 |
| 图片 | 55 GB | **~370 s** | ~5240 s | **~7%** | ⚠️ 可感知 |
| 视频 | 170 GB | **~1130 s** | ~2万 s | **~5–6%** | ⚠️ 可感知 |
| 任意 + TQ | 同上 | 降 >10×（直通+分布式 serdes） | — | **<1%** | ✅ |

### 5.3 网络带宽结论（判据 ρ）

- **文本：ρ<1%** → 数据面完全可忽略，**无必要为经验数据提升任何网络带宽**。
- **图片/视频：ρ≈5–7%**——但这 5–7% 的构成是 **driver 单进程序列化 + Ray 往返**，不是网卡在跑满。**加网络带宽解决不了它**（线上传输只占其中 <5%）；正确解法是 **TQ 式控制面/数据面分离 + worker 分布式直读写**（n_transits 10→~0、serdes 并行化），ρ 压回 <1%。
- 附带收益：driver 单点的 DRAM 汇聚（视频 ~170GB）同时消除（呼应 §2.2）。
- **三代的 scale-out 网卡（25/50GB/s/卡）对数据面全部 ✅**；A2 的 RoCE ❌ 判定是被 all-to-all 借道压垮（§4 的事），与经验数据面无关。

## 6. （下一步）显存带宽：decode 实为 KV-read bound

留作下一步展开，但先给出核心事实——**decode 每步的时间下限由 HBM 读 KV 决定，不由算力/互联决定**：

**per-die HBM BW**：A2 die = A3 die = 1.6 TB/s（A3 封装 3.2 = 2 die），950DT die = 2.0 TB/s（封装 4 ÷ 2）。KV/die 与 rank 数无关（TP4 分片，25.3GB）。

| | KV/die(文本 b=32) | HBM BW/die | **KV 读/步** | 计算/步 | 互联通信/步 |
|---|------------------|-----------|--------------|---------|--------------|
| A2 | 25.3 GB | 1.6 TB/s | **15.8 ms** | 2.4 ms | 7.7 ms（RoCE 借道） |
| A3 | 25.3 GB | **1.6 TB/s** | **15.8 ms** | 2.4 ms | ~1.6 ms（UB per-die） |
| 950DT | 25.3 GB | **2.0 TB/s** | **12.65 ms** | 1.8 ms | ~0.9 ms |

> per-die 修正：**A3 的 KV 读/步从 7.9→15.8ms（= A2）**，因每 die HBM BW 只有 1.6TB/s（封装 3.2 的一半）。**decode 仍是 KV-read bound**（15.8 ≫ 计算 2.4、互联 1.6），且互联通信仍轻松藏进 KV 读窗口 → **判定不变**。A2 与 A3 的 decode 差异只在互联（7.7 RoCE vs 1.6 UB），KV 读相同。

- decode 墙钟 ≈ KV 读时间 ×6.6（vs 计算）——**rollout 生成的真正"带宽诉求"是 HBM 带宽**，而非 scale-up；per-die 下 A2 = A3（都 1.6TB/s），950DT 靠更高 die HBM BW（2.0）领先；
- 杠杆按序：**FP8/INT8 KV**（读量减半）> 提高 HBM BW（代际）> GQA 已用 > 降并发（牺牲吞吐）；视频 KV offload 后每步还要叠 PCIe 读，进一步恶化——是下一步分析的重点；
- 这也解释了 §4 的结论侧影：A3/950DT 上互联通信轻松藏进 KV 读窗口内。

## 7. 汇总矩阵

格式 `A2 / A3 / 950DT`（基线 = P0；各并行配置的同款矩阵见 [§1.5.4](#154-五套配置的-7-式汇总矩阵)）：

| **CPU DRAM 峰值**（含沙箱） | ✅（coding ~156GB/节点） | ✅（~95GB） | ✅~⚠️（465–917GB；最坏堆叠 1.4TB ❌） |

| **scale-out（ρ 判据）** | ✅ ρ<1% | ⚠️ ρ≈7%（软件开销，非带宽） | ⚠️ ρ≈5–6%（同左） |

## 8. 对带宽的诉求（一句话版）

1. **scale-up**：A2 comm-bound（纯前向 τ≈1.3）——但要的是**超节点 fabric**（结构），不是渐进加带宽；**A3 τ≈0.15–0.23、950DT τ≈0.05，全部被计算掩盖，加互联带宽无收益**。
2. **scale-out 网络**：经验数据原始字节秒级搬完，**任何场景都无必要为数据面加网卡带宽**；图片/视频 ρ≈5–7% 的数据面开销是 **driver 序列化 + Ray 静态开销（软件）**，解药是 TQ 直通，不是带宽。
3. **真正值得投资的"带宽"是 HBM 带宽**：decode 每步 ~6.6:1 地被 KV 读支配（per-die A3：KV 读 15.8ms vs 计算 2.4ms；A2 = A3，都 1.6TB/s die）——FP8 KV、更高 HBM BW 代际（950DT die 2.0TB/s）、（视频）避免 KV 落 PCIe，才是 rollout 侧的带宽杠杆（§6，下一步）。
4. **memory-first 并行不抬带宽诉求**（§1.5）：视频放不下的正解是 **开 SP（免费）+ TP↑到 8-die UB 域**——把激活/KV 切进 UB 域、消除 KV offload，**EP τ 不变、A3 全程 ✅**；只有被迫上**跨节点 CP** 时带宽诉求才真正上升，**PP 则是拿 MFU 换内存的训练相开关**。A2 仍需超节点 fabric，非配置可救。
5. **per-die 口径下 HBM 已非区分代际的维度**：A2 与 A3 每 die 完全相同（64GB / 1.6TB/s），视频要放下三代都得 memory-first（P2 + ETP）；**代际唯一实质差异是 scale-up fabric 能否装下 EP=32 的 32-die 域**——A2（8-die 岛）不行、A3/950DT（超节点）行。物理卡数：A2 128 / A3 64 / 950DT 64。

## 9. 方法论边界与假设

- **MFU=40%**（τ 的计算侧）：MFU 提至 60% 时 τ ×1.5，A3 最紧阶段 0.23→0.35 仍 ✅，结论稳健。
- **重叠工程假设**：τ<1 即判"可掩盖"隐含 dispatch/combine 与专家计算重叠（DualPipe 类）已做；无重叠实现下 τ 应对照 1 收紧到 0.5。
- **serdes/Ray 参数**（1.5GB/s、10 次中转、ms 级 RPC）为代表值，实现相关，**需以目标框架实测校准**；TQ 后的 >10× 改善来自 TQ 分析报告的结构推理。
- **沙箱模型**（M_base/M_spike/κ）不可解析，上线前必须压测；κ 对波同步 rollout 取高值是保守但正确的姿态。
- **decode 延迟**（3μs/25μs 单跳）为量级代表值；专家负载不均的 all-to-all 长尾未建模，会让 A2 更差、对 A3/950DT 余量内可吸收。
- 数字均为量级 sizing，换并行度（EP/N/b）需按公式重代入。

## 附录 A · 公式与符号汇总

### A.0 符号定义

**并行度**
- `N` = 总 die 数 = world_size = 128（= TP×CP×PP×DP）
- `TP` / `CP` / `PP` / `DP` = 张量 / 上下文 / 流水线 / 数据 并行度（单位：die）
- `EP` = 专家并行度（= CP×DP）；`SP` = 序列并行（= TP，不额外占卡）；`ETP` = 专家张量并行

**模型结构**
- `S` = 序列长；`P` = prompt 长；`R` = response = S − P
- `h` = 隐藏维 = 4096；`L` = 层数 = 94；`top_k` = 每 token 激活专家数 = 8；`h_kv` = KV 头总维 = 512（GQA）

**批次与采样**
- `B` = 全批轨迹数 = N_p×G = 16384；`N_p` = prompt 数 = 1024；`G` = 每 prompt 采样数 = 16
- `b` = 推理批（每组并发序列数）；`n` = 节点数

**CPU DRAM / 沙箱**
- `C_traj` = 每节点并发轨迹（= 每节点沙箱数）；`f_sb` = 触发沙箱的轨迹比例
- `M_base` / `M_spike` = 单沙箱 稳态 / 峰值 内存；`κ` = 波同步相关系数；`M_shared` = 节点常驻共享（检索索引等）

**带宽与时间**
- `τ` = 通信/可重叠计算；`ρ` = 未掩盖通信/端到端；`MFU` = 算力利用率 ≈ 40%
- `T_comm` / `T_compute` / `T_exposed` / `T_e2e` / `T_dataplane` = 阶段通信 / 可重叠计算 / 未掩盖通信 / 端到端 / 数据面 时间
- `BW` = scale-up 带宽（per-die 196 GB/s uni）；`BW_serdes` = 序列化吞吐；`HBM_BW` = 显存带宽
- `n_transits` = 样本过 driver 中转次数；`V_batch` = 数据面批量字节；`N_rpc` / `t_rpc` = Ray 调用次数 / 单次开销

**精度**：训练 BF16（2B）/ 优化器态；推理权重 FP8（1B）。

### A.1 HBM（per-die 峰值）

```
HBM峰值        = max(训练相, 推理相)                         # 共卡分时

训练相         = 静态 + 激活
  静态         = 专家权重/EP + 非专家ZeRO常驻 + 梯度/N + 优化器/N ≈ 40 GB
  激活         = L × S × h × 2B / (TP×CP)                     # 残差流+激活重算, 每序列

推理相         = 权重底座 + KV
  底座         = 专家(FP8)/EP + 非专家/TP ≈ 9 GB
  KV/序列/die  = 2 × h_kv × L × 2B × S / (TP×CP)              # 2=K,V; GQA h_kv=512
  KV(总)       = KV/序列 × b                                   # b=并发批
     ↳ 校验：文本16k/TP4 = 2×512×94×2×16384/4 = 0.79 GB/序列  ✓

判定           = HBM峰值  ≤  per-die 容量 (A2 64 / A3 64 / 950DT 72)
```

### A.2 CPU DRAM（单节点峰值）

```
节点峰值       = 缓冲区 + KV_offload + 优化器_offload + 沙箱 + OS(~50GB)

B              = N_p × G                                       # 16384 = 1024×16
并发容量       = DP_infer × b ；  波数 = B / (DP_infer × b)
C_traj         = (DP_infer × b) / n                            # 每节点并发轨迹/沙箱数

缓冲区(全集群) = B × S × bytes/token  (+ B × 帧字节)           # bytes/token≈token_id 4B+logprob 2×2B≈8B
缓冲区(节点)   = 缓冲区(全集群) / n

沙箱(节点)     = C_traj × f_sb × [ M_base + κ·(M_spike − M_base) ] + M_shared + 页缓存
优化器_offload = 优化器状态总量 / n                            # 仅紧配置触发, A2=A3
```

### A.3 并行度关系

```
N = TP × CP × PP × DP = 128 (die)      # world_size
EP = CP × DP           SP = TP          # SP 不占卡; EP 骑在 CP×DP 上
专家权重/die = 专家参数 × 精度 / (EP × ETP)    # ETP=1 时专家在 TP 组内复制
训练激活 ∝ 101 / (TP×CP)  [GB, 视频128k 基准]
推理 KV  ∝ 6.3·b / (TP×CP) [GB]
物理卡(封装)数 = N / (每封装 die 数)   # A2 128 / A3 64 / 950DT 64
```

### A.4 带宽判据（相对值）

```
τ = T_comm(该阶段通信) / T_compute(与之独立、可重叠的计算)
    ↳ 分母仅含独立可并发计算, 不含依赖该通信的下游(如 dispatch 后的专家 GEMM)
ρ = T_exposed(未掩盖通信) / T_e2e
    T_e2e = T_rollout + T_logprob + T_train + T_dataplane (+ T_reward)

阈值：τ<0.5 ✅ / 0.5–1 ⚠️ / >1 ❌      ρ<5% ✅ / 5–15% ⚠️ / >15% ❌
τ 对 die/封装口径不变（分子分母同步 ×2）
```

### A.5 scale-up：EP all-to-all 通信量与 τ

```
a2a 单元        = 2 × top_k × h × 2B = 128 KB / token / 层
单卡通信/前向   = 2 × top_k × h × 2B × L = 12.3 MB / token
T_comm          = 通信量 / BW_scale-up(per-die 196 GB/s uni)
T_compute(阶段) = FLOPs / (MFU × 峰值算力 × N)
τ               = T_comm / T_compute
A2 有效 BW      = 32.5 GB/s   # EP=32 跨 4×8-die 岛, ~77% 借道 RoCE 25GB/s
```

### A.6 scale-out：经验数据面与 ρ

```
T_dataplane = n_transits × V_batch / BW_serdes + N_rpc × t_rpc
              ↳ 基线 n_transits≈10, BW_serdes≈1.5GB/s(driver单核); TQ 后 n_transits→0, serdes ×8–16
ρ = T_dataplane / T_e2e
```

### A.7 HBM 带宽：decode（KV-read bound）

```
KV读/步       = KV/die / HBM_BW(per-die: A2/A3 1.6, 950DT 2.0 TB/s)
decode每步墙钟 ≈ max( 计算/步 , KV读/步 , 未掩盖互联/步 )    # 通常 KV 读主导 (~6.6:1)
```

### A.8 长上下文 scaling（拉长 S 时）

```
激活 ∝ S      KV ∝ b·S      MLP 计算 ∝ S      注意力计算 ∝ S²      R = S − P
需 TP×CP  ≳  激活(S) / (die容量 − 静态)          # 决定要开多大 CP
  ↳ CP 是唯一抬高跨节点 scale-up 诉求的杠杆; S↑ → CP↑ → 带宽诉求↑
全注意力适用域：S ≲ 1–2M；S≳5M 时 S² 计算墙 + CP 超卡预算 → 需稀疏注意力 + KV 压缩(另一区间)
```
